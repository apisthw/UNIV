# เข็มทิศ — เอกสารสรุปโปรเจกต์ (Project Brief)

> เอกสารนี้เขียนขึ้นเพื่อให้ทั้ง **คน** และ **AI ที่มาสานงานต่อ** เข้าใจโปรเจกต์นี้ได้แบบไร้รอยต่อ โดยไม่ต้องไล่อ่านบทสนทนาทั้งหมดที่ใช้สร้างมันขึ้นมา อ่านเอกสารนี้ก่อนแก้โค้ดหรือเพิ่มเนื้อหาใดๆ

**สถานะล่าสุด:** ต้นแบบใช้งานได้จริง (functional prototype) — ไม่ใช่แค่ mockup แต่ทำงานได้จริงในเบราว์เซอร์ พร้อมข้อมูลจริงบางส่วนและข้อมูลตัวอย่างที่ระบุไว้ชัดเจน

**ไฟล์เดียวทั้งโปรเจกต์:** `index.html` (ไม่มี dependency ภายนอกอื่นนอกจาก Tailwind CDN, ไม่มี build step, เปิดด้วย double-click ได้เลย)

---

## 1. โปรเจกต์นี้คืออะไร

**เข็มทิศ** เป็นเว็บแอปสำหรับติวและติดตามการเตรียมสอบเข้า **คณะวิศวกรรมศาสตร์** ในระบบ TCAS ของไทย โดยเจาะกลุ่มเป้าหมายเฉพาะ:

- **นักเรียน ม.ปลาย ระดับกลางๆ** (ไม่อ่อน แต่ไม่เก่ง) ที่ต้องการเตรียมตัวสอบเข้าวิศวะอย่างจริงจัง
- **เน้นสายวิศวกรรมไฟฟ้า / คอมพิวเตอร์ / ปัญญาประดิษฐ์ (AI)** เป็นพิเศษ — ไม่ใช่วิศวะทั่วไปแบบกว้างๆ
- เป้าหมายมหาวิทยาลัยหลัก: **3 พระจอมฯ** (มจธ., สจล., มจพ.) บวกมหาวิทยาลัยอื่นที่มีคณะวิศวะ (จุฬาฯ, ธรรมศาสตร์, เกษตรศาสตร์, มหิดล, ศิลปากร, เชียงใหม่, ขอนแก่น, สงขลานครินทร์, สุรนารี)

### สิ่งที่เว็บนี้ทำ 5 อย่าง (5 แท็บหลัก)

| แท็บ | หน้าที่ |
|---|---|
| **ภาพรวม** (Dashboard) | สรุปสถิติผู้เรียน, กำหนดการ TCAS, ประกาศข่าวจริง (คัดเลือก+ตรวจสอบไขว้ด้วยมือ) |
| **ปฏิทิน & เกณฑ์** | เส้นเวลา TCAS, ตารางเปรียบเทียบเกณฑ์คะแนน 12 มหาวิทยาลัย, ศูนย์ลิงก์ทางการ (ของจริง ตรวจสอบแล้ว) |
| **คลังข้อสอบ** | แบบฝึกหัดสไตล์ "ติวเตอร์สอน" — มีมินิบทเรียนก่อนเริ่ม, คำใบ้, เฉลยทีละขั้น, เทคนิคทำข้อสอบ, ผังหัวข้อ (Coverage Map) |
| **บทเรียน** | สรุปเนื้อหา+สูตรต่อหัวข้อ พร้อม checklist ติดตามว่าเรียนหัวข้อไหนแล้วบ้าง |
| **ความคืบหน้า** | มิเตอร์ความพร้อมต่อวิชา, จุดอ่อนที่ควรทบทวน (คำนวณจากข้อมูลจริงที่ผู้ใช้ทำในเครื่อง) |

## 2. แนวคิดและหลักการออกแบบ

### ชื่อและธีม
- ชื่อโปรเจกต์: **"เข็มทิศ"** (เดิมชื่อ "วงเวียน" — เปลี่ยนชื่อระหว่างพัฒนา ดู §9 เรื่อง legacy naming ที่ยังหลงเหลือใน localStorage keys)
- Subtitle: "นำทางสู่คณะวิศวกรรมศาสตร์"
- ธีมภาพ: **โต๊ะเขียนแบบวิศวกรรม (engineering drafting board)** — ใช้ blueprint grid จางๆ เป็นพื้นหลัง, การ์ดมีขีดมุมแบบ registration mark (มุมซ้ายบน-ขวาล่าง), โทนสีส้ม "drafting pencil" เป็นสี accent หลัก
- โลโก้: วงกลม + เส้นรัศมี + จุดศูนย์กลาง — อ่านได้ทั้งเป็น "วงเวียนเขียนแบบ" และ "เข็มทิศนำทาง" (ตั้งใจให้กำกวมแบบมีเหตุผล จะได้ไม่ต้องแก้ไอคอนตอนเปลี่ยนชื่อ)

### หลักการที่ยึดตลอดทั้งโปรเจกต์ (สำคัญมาก — อ่านก่อนเพิ่มเนื้อหาใดๆ)

1. **ห้ามเสนอข้อมูลปลอมเป็นข้อมูลจริงเด็ดขาด** — สัดส่วนคะแนนเปรียบเทียบ (`compareData`) ยังเป็น**ข้อมูลตัวอย่าง**ที่ระบุไว้ชัดเจน (มีแบนเนอร์เตือนถาวรในหน้า Dashboard และ Calendar) เพราะดึงข้อมูลสดจากเว็บทางการไม่ได้จริง (ไม่มี public API, บางเว็บบล็อก iframe, และเป็น static file ไม่มี backend) — กำหนดการสอบ (`rounds`/`examDates`) และข่าวประกาศ (`news`) เปลี่ยนจากข้อมูลตัวอย่างเป็น**ข้อมูลจริงที่ค้นคว้าและตรวจสอบไขว้แล้ว** ตั้งแต่ 12–13 ก.ค. 2569 (ดู §13 ข้อ 71, 72) แต่เป็นภาพนิ่ง ณ วันที่ค้นคว้า ไม่ใช่ระบบดึงข่าวสดอัตโนมัติ — ต้องระบุแหล่งที่มาและวันที่กำกับเสมอ ห้ามลบ disclaimer ที่ชี้ให้ผู้ใช้ไปตรวจสอบซ้ำกับแหล่งทางการ
2. **ข้อสอบทุกข้อเป็น "แบบฝึกหัดจำลอง" ไม่ใช่ข้อสอบจริงที่หลุดออกมา** — ระบุไว้ในหัวแท็บคลังข้อสอบชัดเจน เนื้อหาถูกต้องตามหลักวิชา แต่ไม่ได้อ้างว่าเป็นข้อสอบปีใดปีหนึ่งจริง — **ตั้งแต่ 12 ก.ค. 2569 เนื้อหาทุกวิชาผ่านการตรวจสอบสไตล์เทียบกับตัวอย่างข้อสอบจริงจากสถาบันติวที่มีชื่อเสียงแล้ว** (ไม่ใช่แค่คิดขึ้นเองตามความเข้าใจอย่างที่เคยเป็นมาก่อน — ดู §6.2 สำหรับเรื่องราวเต็มและ §12.1 สำหรับแหล่งอ้างอิงต่อวิชา) แต่ยังคงเป็นโจทย์ที่เขียนขึ้นใหม่ ไม่ใช่ข้อสอบจริงที่คัดลอกมา
3. **ห้ามเดา/แต่งลิงก์เว็บไซต์ทางการ** — ทุก URL ใน `officialLinks` ผ่านการค้นหาและตรวจสอบด้วย WebSearch จริงก่อนใส่ (ไม่ใช่เดาจากความจำ) ดู §12
4. **ห้ามสร้างเนื้อหาที่ตรวจสอบความถูกต้องไม่ได้** — เหตุผลที่ "มิติสัมพันธ์ (Diagrammatic/Spatial Reasoning)" ของทั้ง TPAT3 และ TGAT2 ยังไม่มีเนื้อหา เพราะต้องใช้โจทย์ภาพ/แผนภาพหมุนที่สภาพแวดล้อมการพัฒนานี้**ไม่มีเครื่องมือแคปหน้าจอ/ดู SVG render จริง** จึงไม่กล้าเดาส่งเพราะเสี่ยงผิดแบบตรวจไม่พบ — นี่คือหลักการ ไม่ใช่แค่ "ยังไม่ว่าง"
5. **ผังหัวข้อ (topicMap) ต้องซื่อสัตย์เสมอ** — สถานะ `ready` / `partial` / `none` ต้องตรงความจริง ห้ามอวยว่า "มีครบ" ถ้ายังไม่ครบจริงตาม Test Blueprint ทางการ
6. **ทุกครั้งที่แก้โค้ด ต้อง trace ผลกระทบรอบด้าน (360°) ก่อนแก้ และตรวจสอบหลังแก้เสมอ (Post-deploy verification)** — กฎนี้ตั้งขึ้นโดยผู้ใช้ (12 ก.ค. 2569) หลังรอบ refactor ใหญ่ (แยก `data/*.js`) ที่การเรียงลำดับโหลด/การอ้างอิงข้าม field ผิดพลาดได้ง่ายโดยไม่มีอาการเตือนทันที รายละเอียดวิธี trace และตรวจสอบอยู่ใน §10

## 3. สถาปัตยกรรมทางเทคนิค

```
d:\MyData_Lenovo P14s\Web_UNIV\
├── index.html              ← โครง HTML + CSS + ฟังก์ชัน JS ทั้งหมด (ไม่มีข้อมูลวิชาฝังอยู่แล้ว)
├── data/                   ← บทเรียน+ชุดข้อสอบ แยกไฟล์ตามวิชา (เพิ่ม 12 ก.ค. 2569 — ดู §13 ข้อ 19)
│   ├── tpat3.js
│   ├── alevel-math1.js
│   ├── alevel-phys.js
│   ├── alevel-chem.js
│   ├── alevel-math2.js
│   ├── alevel-eng.js
│   ├── tgat.js
│   └── enrichment.js
└── README.md               ← เอกสารนี้
```

- **ไม่มี build step, ไม่มี package.json, ไม่มี node_modules, ไม่มี bundler** — เปิด `index.html` ด้วยเบราว์เซอร์ได้ตรงๆ แม้แบบ double-click (`file://`)
- **Tailwind CSS ผ่าน CDN** (`<script src="https://cdn.tailwindcss.com">`) — เลือกใช้ CDN เพราะไฟล์นี้เป็นไฟล์จริงที่เปิดจาก `file://` ในเบราว์เซอร์ปกติ ไม่ใช่ Artifact แบบ sandbox (Artifact จะบล็อก CDN ด้วย CSP แต่ที่นี่ไม่ติดปัญหานั้น)
- **ข้อมูลวิชาแยกไฟล์ (`data/*.js`) โหลดด้วย `<script src="data/xxx.js">` ธรรมดา ไม่ใช่ `fetch()`** — จุดนี้สำคัญ: `fetch()` JSON จาก `file://` จะโดน CORS บล็อกใน Chrome แต่ `<script src>` ไม่ติดปัญหานี้ จึงยังเปิดแบบ double-click ได้เหมือนเดิม แต่ละไฟล์ทำ `window.KEMTID_DATA.lessons.push(...)` และ `window.KEMTID_DATA.quizzes.push(...)` เข้า object กลาง — ต้องโหลดไฟล์เหล่านี้ *ก่อน* `<script>` หลักเสมอ (เรียงลำดับใน `index.html` แล้ว) เพราะสคริปต์หลักอ่าน `const lessons = window.KEMTID_DATA.lessons;` ตรงๆ
- **สาเหตุที่แยกไฟล์ (ไม่ใช่ single-file เหมือนก่อนหน้านี้):** ผู้ใช้แจกจ่ายเว็บนี้เป็น zip ทั้งโฟลเดอร์อยู่แล้ว (ไม่ได้ต้องพึ่งไฟล์เดียว) จึงเปลี่ยนมาแยกไฟล์ตามวิชาเพื่อให้แก้ไข/ขยายเนื้อหาแต่ละวิชาได้ง่ายขึ้นโดยไม่ต้องไล่หาตำแหน่งในไฟล์รวมที่ยาวขึ้นเรื่อยๆ (ดู §13 ข้อ 19 สำหรับเหตุผลเต็ม)
- **Vanilla JavaScript** ฟังก์ชัน/logic ทั้งหมดอยู่ใน `<script>` เดียวท้ายไฟล์ ห่อด้วย IIFE `(function () { "use strict"; ... })();` (ข้อมูลวิชาอยู่นอก IIFE นี้ ใน `data/*.js`)
- **ไม่มี framework** (ไม่ใช้ React/Vue) — render ด้วยการต่อ string HTML แล้วยัดเข้า `.innerHTML` ตรงๆ
- **Persistence:** `localStorage` เท่านั้น (ไม่มี backend/server) — ข้อมูลอยู่เฉพาะเครื่อง/เบราว์เซอร์ที่ใช้งาน

### ⚠️ กับดักทางเทคนิคที่ต้องรู้ (Tailwind + CSS custom properties)

ระบบสีทั้งหมดคุมด้วย **CSS custom properties** ใน `:root` และ `html.dark` (สลับ light/dark ด้วยการ toggle class `dark` บน `<html>`) แล้วให้ `tailwind.config` ชี้ไปที่ตัวแปรเหล่านั้น

**กฎสำคัญ:** สี Tailwind ที่ต้องใช้กับ opacity modifier (เช่น `bg-accent/10`, `border-good/40`) **ต้องประกาศเป็น `rgb(var(--x-rgb) / <alpha-value>)`** โดยมี custom property แยกต่างหากเป็นรูป `"R G B"` (ตัวเลขคั่นด้วยเว้นวรรค ไม่ใช่ hex) เช่น:

```css
--accent-rgb: 224 129 47;   /* ไม่ใช่ --accent: #e0812f แล้วหวังว่า bg-accent/10 จะทำงาน */
```
```js
accent: 'rgb(var(--accent-rgb) / <alpha-value>)'
```

ถ้าประกาศเป็น `var(--accent)` (hex ตรงๆ) แล้วใช้ `/10` opacity modifier กับมัน **จะไม่ทำงาน** — นี่คือบั๊กที่เจอจริงหลายรอบระหว่างพัฒนา (ต้องแก้ทีหลังสำหรับ `surface`, `accent2` ด้วย) สีที่มี rgb-pair แล้วตอนนี้: `accent`, `accent2`, `good`, `warn`, `bad`, `neutral`, `surface` — สีอื่น (`bg`, `surface2`, `line`, `ink`, `ink2`, `ink3`, `accentink`) เป็น hex ตรงๆ เพราะไม่เคยใช้ opacity modifier กับมัน **ถ้าจะเพิ่มสีใหม่ที่ต้องใช้ `/opacity` ให้ทำตามแพทเทิร์น rgb-pair นี้เสมอ**

### Design tokens (สรุปสั้น)
- ฟอนต์: `Sarabun` (ไทย, body/heading) + `JetBrains Mono` (ตัวเลข/label เทคนิค, ใช้ `font-tabular` สำหรับเลขคอลัมน์)
- Accent: ส้ม drafting-pencil (`#c1571f` light / `#e0812f` dark)
- Semantic status: `good` (เขียว), `warn` (เหลือง/น้ำตาล), `bad` (แดง), `neutral` (เทา) — ใช้กับ pill สถานะ (เปิดรับสมัคร/ปิดรับแล้ว, ระดับความแม่นยำ, coverage status)
- Corner ticks (`.tick` class): มุมซ้ายบน-ขวาล่างมีเส้นสีส้มเล็กๆ แบบ registration mark — ใช้กับการ์ดสำคัญเท่านั้น ไม่ใช่ทุกการ์ด

## 4. โครงสร้างข้อมูล (Data Model)

### ตำแหน่งไฟล์ของแต่ละส่วนข้อมูล

| ข้อมูล | อยู่ที่ไฟล์ไหน |
|---|---|
| `lessons[]`, `quizzes[]` (เนื้อหารายวิชา) | **`data/{subjectId}.js`** — แยกไฟล์ตามวิชา (8 ไฟล์) push เข้า `window.KEMTID_DATA` |
| `subjects[]`, `topicMap{}`, `officialLinks[]`, `rounds[]`, `examDates[]`, `compareData[]`, `news[]`, `majors[]` | `index.html` — ต้นๆ ของ `<script>` หลัง `<script src="data/*.js">` ทั้ง 8 บรรทัด |

ใน `index.html` สคริปต์หลักอ่านข้อมูลวิชาแบบนี้ (ไม่ได้ประกาศ array ตรงๆ อีกต่อไป):
```js
const lessons = window.KEMTID_DATA.lessons;
const quizzes = window.KEMTID_DATA.quizzes;
```

### `subjects[]` — รายวิชา/สนามสอบ (8 รายการ, อยู่ใน `index.html`)
```js
{ id: 'tpat3', name: 'TPAT3 · ความถนัดวิทยาศาสตร์ เทคโนโลยี วิศวกรรมศาสตร์', short: 'TPAT3' }
```
รายการ id ทั้งหมด: `tpat3`, `alevel-math1`, `alevel-phys`, `alevel-chem`, `alevel-math2`, `alevel-eng`, `tgat`, `enrichment` — **id เหล่านี้ตรงกับชื่อไฟล์ใน `data/` เป๊ะๆ** (เช่น `alevel-phys` → `data/alevel-phys.js`)

`comingSoon` (Set): ว่างเปล่าแล้ว (ทุกวิชามีเนื้อหาอย่างน้อยบางส่วน) — วิชาที่ไม่มีเนื้อหาเลยจะถูกใส่ id ไว้ใน Set นี้เพื่อโชว์การ์ด "เร็วๆ นี้" แทน

### `lessons[]` — บทเรียน (32 รายการ, ใน `data/{subjectId}.js`)
```js
{
  id: 'l1', subjectId: 'alevel-math1', title: 'ฟังก์ชันและกราฟ',
  body: '...', formula: 'f(x) = ax² + bx + c → ...',
  detail: {
    concept: 'แก่นความคิดหลักของบทนี้ 1-2 ประโยค',
    explanation: 'คำอธิบายเต็มที่เชื่อมโยงสูตร/หลักการเข้าด้วยกัน มักโยงไปยังการใช้งานจริง',
    examples: [
      { q: 'โจทย์ตัวอย่างที่ 1', steps: ['ขั้นที่ 1', 'ขั้นที่ 2'], answer: 'คำตอบ' },
      { q: 'โจทย์ตัวอย่างที่ 2', steps: ['...'], answer: '...' }
    ]
  }
}
```
- `formula: '—'` หมายถึงบทเรียนนั้นไม่มีสูตรสรุป (เช่นบทที่เป็นแนวคิด/นิยามล้วน)
- `detail` (เพิ่ม 12 ก.ค. 2569, ขยาย `explanation`+`examples` 13 ก.ค. 2569) คือเนื้อหาสรุปสำหรับทบทวน แสดงในพาเนล "ดูสรุปบทเรียน" ที่ขยาย/ย่อได้ในหน้าบทเรียน (ดู §5) — **ต้องมีครบทุกบท และ `examples` ต้องมีครบ 5 ข้อ/บท** (ตรวจสอบด้วยสคริปต์ §10 ว่าไม่มีบทไหนขาด `detail`/`examples` หรือมี `examples` ไม่ครบ 5) — โครงสร้างเลียนแบบ `quiz.intro` (concept + worked) โดยตั้งใจให้สอดคล้องกัน เพียงแต่ `examples` เป็นอาเรย์ 5 ข้อ (ไม่ใช่ตัวอย่างเดียวแบบ `quiz.intro.worked`) เพราะเจตนาใช้ทบทวนซ้ำหลายมุมมอง ไม่ใช่แค่ปูพื้นก่อนทำโจทย์ — `explanation` ทุกบทเขียนด้วยรูปแบบ "ลองนึกภาพ..." (ภาพเปรียบเทียบที่จับต้องได้ก่อนอธิบายสูตร) เพื่อให้เห็นภาพง่ายขึ้นโดยไม่เพิ่มความซับซ้อน
- id ไม่เรียงต่อเนื่องสมบูรณ์ทั้งระบบ (มี l6b, l6c, m2_1-5, eng_1-2 ฯลฯ เพราะเพิ่มทีหลังในหลายรอบ) — **อย่ายึด id เป็นลำดับ ให้ยึด `subjectId` แทน** — ภายในไฟล์เดียวกัน (วิชาเดียวกัน) ก็ไม่จำเป็นต้องเรียงต่อเนื่องเช่นกัน แค่ห้ามซ้ำกับ id ที่มีอยู่แล้วทั้งระบบ (ตรวจด้วยสคริปต์ §10 เสมอ)

### `quizzes[]` — ชุดข้อสอบ (88 ชุด, ใน `data/{subjectId}.js`)
```js
{
  id: 'q_alevel_phys_1', subjectId: 'alevel-phys',
  title: 'A-Level ฟิสิกส์ · ชุดฝึกหัด 1', topic: 'กลศาสตร์และวงจรไฟฟ้าเบื้องต้น', difficulty: 'กลาง',
  intro: {
    concept: 'ข้อความสอนแนวคิดหลักก่อนเริ่มทำโจทย์ (2-4 ประโยค)',
    worked: { q: 'โจทย์ตัวอย่าง (ไม่ซ้ำกับข้อในชุด)', steps: ['ขั้นที่ 1...', 'ขั้นที่ 2...'], answer: 'คำตอบสรุป' },
  },
  questions: [
    { q: 'คำถาม', choices: ['ตัวเลือก1','ตัวเลือก2','ตัวเลือก3','ตัวเลือก4'], correct: 1 /* index */,
      hint: 'คำใบ้ก่อนตอบ (Socratic — ชวนคิดไม่บอกคำตอบ)',
      steps: ['ขั้นตอนวิธีคิดทีละขั้น', '...'],
      tip: 'เทคนิคทำข้อสอบ (ไม่ใช่แค่คำอธิบายเนื้อหา)' },
  ],
}
```
**ทุก field เป็น required ในทางปฏิบัติ** — โค้ด render (`playQuiz()`) เรียก `item.hint`, `item.steps`, `item.tip` ตรงๆ โดยไม่มี fallback ถ้าขาด field จะเป็น `undefined` โผล่ในหน้าเฉลย

### `topicMap{}` — ผังหัวข้อเทียบกับ Test Blueprint จริง (สำคัญที่สุดสำหรับวางแผนงานต่อ)
```js
'alevel-phys': {
  total: '30 ข้อ (25 ปรนัย + 5 อัตนัย) · 90 นาที',
  sections: [
    { name: 'กลศาสตร์', items: [
      { name: 'การเคลื่อนที่ แรงและกฎนิวตัน ...', count: '8–10 ข้อ', status: 'partial' }
    ]},
  ],
}
```
- `status` มี 4 ค่า: `'ready'` (ยังไม่เคยใช้จริงในโปรเจกต์นี้ — ไม่มีหัวข้อไหนถึงระดับ "มีครบ" ตาม blueprint จริง), `'partial'` (มีเนื้อหาบางส่วนในเว็บนี้เอง), `'none'` (ยังไม่มีเลย ไม่มีแม้แต่ลิงก์แนะนำ), `'external'` (ไม่มีชุดฝึกในเว็บนี้ แต่มีลิงก์ไปแหล่งฝึกภายนอกที่ตรวจสอบแล้วใน `officialLinks` — ใช้กับ "มิติสัมพันธ์" ของ TPAT3/TGAT2 ที่ต้องการสื่อภาพซึ่งสร้างเองไม่ได้ในสภาพแวดล้อมนี้)
- คีย์ของ object ต้องตรงกับ `subjects[].id` — เฉพาะวิชาที่มี entry ใน `topicMap` เท่านั้นที่จะโชว์ในการ์ด "ผังหัวข้อทั้งหมด" (แท็บคลังข้อสอบ) — `enrichment` **ไม่มี** entry ใน topicMap โดยตั้งใจ เพราะไม่ใช่สนามสอบจริง

## 5. Features ทั้งหมด (รายละเอียดเชิงพฤติกรรม)

### Dashboard
- Stat tiles: จำนวนข้อสอบที่ทำ, ความแม่นยำเฉลี่ย, บทเรียนที่จบ (X/32), เป้าหมาย (ข้อความ static)
- การ์ดกำหนดการ 3 รอบถัดไป (คำนวณสถานะสดจากวันที่ปัจจุบัน)
- รายการข่าวจริง 3 รายการ (ค้นคว้า+ตรวจสอบไขว้แล้ว) พร้อมลิงก์แหล่งข่าวต่อรายการ และปุ่มลิงก์ภายนอกไปหน้าข่าว mytcas.com เต็ม

### ปฏิทิน & เกณฑ์
- เส้นเวลาเต็ม 5 รอบ
- ตารางเปรียบเทียบ 12 มหาวิทยาลัย (overflow-x scroll บนจอเล็ก)
- **ศูนย์ลิงก์ทางการ** — การ์ดลิงก์จริง เปิดแท็บใหม่ (`target="_blank" rel="noopener noreferrer"`)

### คลังข้อสอบ (ซับซ้อนที่สุด)
- การ์ดแนะนำตัว "ติวเตอร์เข็มทิศ"
- **คลังข้อที่ทำผิดบ่อย** (`mistakeBank`) — เก็บ key `quizId#questionIndex` อัตโนมัติเมื่อตอบผิด, ลบออกเมื่อตอบถูกในภายหลัง, ปุ่ม "ฝึกข้อที่เคยพลาด" รวมข้อจากทุกวิชามาเป็นชุดพิเศษ (`openMistakeReview()` → สร้าง quiz object สังเคราะห์ที่มี `subjectId: null`)
- **ผังหัวข้อทั้งหมด (Coverage Map)** — toggle แสดง/ซ่อน, render จาก `topicMap`
- **โหมดจับเวลาสอบจริง (Mock Exam)** — เลือกวิชา (`mockSubject`, ค่าเริ่มต้น "รวมทุกวิชาที่มี") + จำนวนข้อ (`mockCount`: 5/10/15/20) → `startMockExam()` สุ่มโจทย์จากทุกชุดที่ตรงเงื่อนไขด้วย `getQuestionPool()` + `shuffle()` (Fisher–Yates) แล้วเรียก `playQuiz(quiz, { timed: true, timeLimitSec })` โดยประเมินเวลาแบบ **90 วินาที/ข้อ** (ค่าคงที่ `SEC_PER_QUESTION`) — โหมดนี้ซ่อนปุ่ม "ขอคำใบ้" ทั้งหมด, มี countdown timer สีเปลี่ยนเป็น warn/bad เมื่อเหลือเวลา ≤30%/≤15%, และ**ส่งคำตอบอัตโนมัติเมื่อหมดเวลา** (`requestSubmit()`) ผลสอบยังบันทึกลง `state.attempts` ภายใต้ id สังเคราะห์ `mock_{timestamp}` (ไม่ชนกับ id ข้อสอบจริง จึงไม่กระทบค่าความแม่นยำรายวิชาใน `accuracyPct()` ซึ่งอ้างอิงจาก `quizzes[]` เท่านั้น) — ข้อที่ตอบผิดยังเข้าคลังข้อที่ทำผิดบ่อยตามปกติ
- Filter ตามวิชา (chip buttons)
- **โหมดทำโจทย์ (`playQuiz(quiz, opts)`)** — `opts.timed`/`opts.timeLimitSec` เปิดโหมดจับเวลา (ดูด้านบน) ปกติเรียกแบบไม่มี `opts`:
  1. มินิบทเรียนก่อนเริ่ม (จาก `quiz.intro`) — ย่อ/ขยายได้
  2. ต่อคำถาม: ปุ่ม "ขอคำใบ้" (toggle แสดง/ซ่อน `item.hint`) ก่อนตอบ
  3. ส่งคำตอบ → ไฮไลต์ถูก/ผิดทุกข้อ, โชว์ "วิธีคิด" เป็น ordered list จาก `item.steps` + เทคนิคติวเตอร์จาก `item.tip`
  4. คำนวณคะแนน% → โชว์ "สรุปจากติวเตอร์" (ข้อความให้กำลังใจ/คำแนะนำตามช่วงคะแนน 4 ระดับ: ≥90%, ≥70%, ≥50%, <50%) + ปุ่มพาไปทบทวนบทเรียนวิชานั้น (`goToLessonsFor(subjectId)` — เลื่อนหน้าไปยัง lesson group ด้วย `scrollIntoView`)
  5. บันทึกลง `state.attempts` และอัปเดต `state.mistakes` อัตโนมัติ

### บทเรียน
- พาเนล "เส้นทางสายไฟฟ้า–คอมพิวเตอร์–AI" (จาก `majors[]`) ด้านบนสุด
- Lesson groups ต่อวิชา (เฉพาะวิชาที่ไม่อยู่ใน `comingSoon`) — แต่ละใบมี checkbox แยกต่างหาก (ไม่ครอบทั้งการ์ดแบบเดิม) ติ๊กว่าทบทวนแล้ว (บันทึกลง `state.lessonsDone`)
- **สรุปบทเรียนสำหรับทบทวน (`lesson.detail`, เพิ่ม 12 ก.ค. 2569, ขยาย 13 ก.ค. 2569)** — ทุกบทเรียน (ครบทั้ง 32 บท) มีปุ่ม "ดูสรุปบทเรียน" ที่ขยาย/ย่อพาเนลแสดง: **แก่นของเรื่อง** (concept แก่นความคิดหลัก 1-2 ประโยค), **คำอธิบาย** (explanation — เขียนแบบ "ลองนึกภาพ..." เปิดด้วยภาพเปรียบเทียบที่จับต้องได้เสมอ เช่น ไม้กระดกแทนคานงัด, เครื่องหยอดเหรียญแทนฟังก์ชัน, ท่อน้ำแทนวงจรไฟฟ้า ก่อนโยงเข้าสูตร/หลักการจริง มักเชื่อมไปยังการใช้งานจริงในวิศวกรรมไฟฟ้าเมื่อเกี่ยวข้อง), และ **ตัวอย่างประกอบ 5 ข้อ/บท** (`examples[]` แต่ละข้อมี `q`/`steps`/`answer` แบบเดียวกับ "วิธีคิด" ในโหมดทำโจทย์ — เพิ่มจาก 2 เป็น 5 ข้อ/บทตามคำขอผู้ใช้ เพื่อให้ทบทวนซ้ำได้หลากหลายมุมมองมากขึ้น) — แก้ปัญหาที่ผู้ใช้ชี้ว่าเดิมมีแค่หัวข้อ+checkbox ซึ่งไม่มีประโยชน์ในการทบทวนจริง ดู §13 changelog ล่าสุด

### ความคืบหน้า
- Readiness score รวม = `lessonPct×0.4 + accuracyPct×0.6` เฉลี่ยข้ามทุกวิชา
- Progress bar ต่อวิชา + จุดอ่อนที่ควรทบทวน (วิชาที่ accuracy <50% หรือ lesson completion <50%)
- **Export/Import ความคืบหน้า** — ปุ่ม "บันทึกเป็นไฟล์" (`exportProgress()`) ดาวน์โหลด `state` ทั้งก้อนเป็น JSON (`kemtid-progress-YYYY-MM-DD.json`); ปุ่ม "โหลดจากไฟล์" (`importProgress()`) อ่านไฟล์ JSON กลับเข้า `state` — มี `window.confirm()` เตือนก่อนเสมอเพราะเป็นการแทนที่ข้อมูลปัจจุบันทั้งหมด (ไม่ได้ merge) แก้ปัญหา localStorage ผูกกับเบราว์เซอร์เดียว

## 6. การแก้ไขครั้งสำคัญที่ต้องรู้: ระบบสอบเปลี่ยนชื่อไปแล้ว

**บริบทสำคัญ:** ประเทศไทยเปลี่ยนระบบสอบ TCAS จาก **GAT/PAT/วิชาสามัญ** เป็น **TGAT/TPAT/A-Level** ตั้งแต่ **TCAS66 (ปีการศึกษา 2566)** เป็นต้นมา

ระหว่างพัฒนาโปรเจกต์นี้ ช่วงแรกใช้ชื่อเก่า ("PAT1", "PAT3") ผิดพลาด เพราะอ้างอิงจากความรู้เดิมโดยไม่ได้ตรวจสอบ ภายหลังใช้ WebSearch ตรวจสอบกับ Test Blueprint จริงจาก `mytcas.com` แล้วพบว่า:

- **"PAT1" ไม่มีอยู่แล้ว** — เนื้อหาความถนัดคณิตศาสตร์ถูกรวมเข้า **A-Level คณิตศาสตร์ประยุกต์ 1/2**
- **"PAT3" → ปัจจุบันคือ TPAT3** และ**โครงสร้างข้อสอบต่างไปจากเดิมโดยสิ้นเชิง** — ไม่ใช่แค่เปลี่ยนชื่อ TPAT3 จริงมี 70 ข้อ แบ่งเป็นด้านความถนัด (ตัวเลข/มิติสัมพันธ์/กลศาสตร์) และด้านความคิด-ความสนใจวิทยาศาสตร์ ซึ่งเป็นโจทย์แนวเหตุผล/อัตนัยเชิงความคิด **ไม่ใช่โจทย์ฟิสิกส์สูตรล้วนแบบที่เข้าใจผิดตอนแรก**

**สิ่งที่ทำไปเพื่อแก้:**
1. เปลี่ยนชื่อ subject/quiz ทุกจุดจาก PAT1/PAT3 → `alevel-math1`/`tpat3`
2. ย้ายเนื้อหาฟิสิกส์ล้วนๆ (กลศาสตร์, ไฟฟ้ากระแสตรง) ที่เคยอยู่ผิดที่ใน "PAT3" ไปไว้ที่ `alevel-phys` (ถูกที่แล้ว เพราะเป็นเนื้อหา A-Level ฟิสิกส์จริง)
3. ย้ายเนื้อหาที่ไม่ใช่หลักสูตรจริงเลย (ไฟฟ้ากระแสสลับเชิงลึก + ตรรกะดิจิทัล) ไปไว้ที่ subject ใหม่ `enrichment` พร้อม label "นอกเนื้อหาสอบ" ชัดเจน
4. สร้างเนื้อหา TPAT3 ใหม่ทั้งหมดให้ตรงกับรูปแบบข้อสอบจริง (ดู §7)

**บทเรียนสำหรับ AI ที่มาทำงานต่อ:** อย่าเชื่อความรู้เดิมเกี่ยวกับชื่อ/รูปแบบข้อสอบไทยโดยไม่ตรวจสอบ ระบบการศึกษาไทยเปลี่ยนบ่อย ให้ WebSearch เทียบกับ `mytcas.com/blueprint/` เสมอก่อนอ้างอิงชื่อสนามสอบหรือโครงสร้างข้อสอบใดๆ

### 6.1 บทเรียนอีกข้อ: WebFetch แปลงปี พ.ศ.→ค.ศ. ผิดพลาดไม่คงเส้นคงวา

ตอนอัปเดต `rounds[]` เป็นปฏิทิน TCAS70 จริง (ก.ค. 2569) พบว่า **เครื่องมือ WebFetch แปลงปี พ.ศ. เป็น ค.ศ. ผิดและไม่ตรงกันเองระหว่างการเรียกครั้งต่างๆ** — ตัวอย่างจริง: วันสอบ TGAT/TPAT ที่ต้นฉบับระบุ "13-15 ธ.ค. 2568" ถูกแปลงเป็นทั้ง "2025" ในครั้งหนึ่งและ **"2024"** ในอีกครั้ง (ผิดทั้งคู่ในทางใดทางหนึ่ง เพราะ 2568-543=2025 ที่ถูกต้อง)

**วิธีแก้ที่ใช้ได้ผล:** สั่งให้ WebFetch **"คงตัวเลขปี พ.ศ. ตามต้นฉบับ ห้ามแปลงเป็น ค.ศ. เอง"** ในพรอมป์อย่างชัดเจน แล้วแปลง พ.ศ.→ค.ศ. **ด้วยตัวเอง** (สูตร: ค.ศ. = พ.ศ. − 543) หลังจากได้ตัวเลขกลับมา — วิธีนี้ให้ผลตรงกันข้ามหลายแหล่งที่ตรวจสอบไขว้กัน ส่วนวัน/เดือนมักไม่มีปัญหาการแปลง (ปัญหาอยู่ที่ปีเท่านั้น)

**บทเรียนสำหรับ AI ที่มาทำงานต่อ:** เมื่อดึงวันที่จากเว็บไทยผ่าน WebFetch **ห้ามเชื่อปี ค.ศ. ที่เครื่องมือแปลงมาให้ทันที** ให้ขอปี พ.ศ. ต้นฉบับมาเสมอแล้วแปลงเอง และควรตรวจสอบไขว้อย่างน้อย 2 แหล่งอิสระก่อนใส่ลงข้อมูลจริงในเว็บ

### 6.2 บทเรียนสำคัญที่สุด: "ตรวจสอบโครงสร้าง" ไม่เท่ากับ "ตรวจสอบสไตล์โจทย์"

ระหว่างสร้างเนื้อหาข้อสอบทั้งหมด (§7 เดิม 234 ข้อ) มีการตรวจสอบแค่ **"โครงสร้าง"** ของแต่ละวิชา (จำนวนข้อ, สัดส่วนคะแนน, ชื่อหมวดหมู่) กับ Test Blueprint ทางการจาก mytcas.com เท่านั้น — ส่วน **"เนื้อหาโจทย์จริง"** (ตัวเลขในโจทย์, สไตล์การตั้งคำถาม, ประเภทของโจทย์ในแต่ละหมวด) เขียนขึ้นเองทั้งหมดจากความรู้เนื้อหาวิชา **ไม่เคยเทียบกับตัวอย่างโจทย์จริงจากสถาบันติวหรือแหล่งข้อสอบที่มีชื่อเสียงเลยแม้แต่ข้อเดียว**

ผู้ใช้ตั้งคำถามตรงจุดนี้เมื่อ 12 ก.ค. 2569 ("ออกแบบ อ้างอิง มาจากสถาบันติว ติวเตอร์ที่ประสบความสำเร็จ...หรือไม่") — เมื่อตรวจสอบจริง (WebSearch/WebFetch เทียบกับ SmartMathPro, AppliedPhysics สำหรับ TPAT3) **พบว่าสไตล์โจทย์ไม่ตรงกับของจริงในหลายจุดที่สำคัญ** เช่น TPAT3 ด้านตัวเลขของจริงมีโจทย์แนว "นิยามการดำเนินการขึ้นเอง" (เช่น `a@b = 2a+b` แล้วให้หาค่าใหม่) เป็นสัดส่วนสำคัญ ซึ่งเนื้อหาเดิมไม่มีเลยแม้แต่ข้อเดียว (เป็น word problem ทั้งหมด) — ดู §7 ตารางสำหรับรายละเอียดเพิ่มเติมของแต่ละวิชาที่รื้อสร้างแล้ว

**บทเรียนสำหรับ AI ที่มาทำงานต่อ:** การตรวจสอบ "โครงสร้าง/จำนวนข้อ/หมวดหมู่" กับ blueprint ทางการ **ไม่เพียงพอ** สำหรับยืนยันว่าโจทย์ที่เขียนขึ้นเอง "ใช้งานได้จริง" — ต้องหาตัวอย่างโจทย์จริง (verbatim) จากแหล่งที่มีชื่อเสียง/มีสถิติเด็กสอบติดจริง (เช่น SmartMathPro, OpenDurian, Dek-D, Ondemand, AppliedPhysics, Chulatutor) มาเทียบสไตล์ก่อนเขียนเนื้อหาใหม่ทุกครั้ง ไม่ใช่แค่ตรวจสอบว่า "หมวดหมู่ถูกไหม" แล้วเขียนโจทย์เองตามความเข้าใจ — ข้อจำกัดที่แก้ไม่ได้จริงๆ (เช่น หมวด "ความสนใจข่าวสาร" ที่ต้องการความรู้ข่าวเฉพาะช่วงเวลา) ให้ระบุไว้ตรงๆ ใน `intro.concept` ของชุดข้อสอบนั้น อย่าเดาข่าวปลอมหรือแกล้งทำเป็นข่าวปัจจุบันที่ตรวจสอบไม่ได้

## 7. สถานะเนื้อหาปัจจุบัน (ณ วันที่เขียนเอกสารนี้)

**รวม: 32 บทเรียน, 119 ชุดข้อสอบ, 472 ข้อ** (ตัวเลขนี้ต้องตรวจสอบด้วยสคริปต์ Node ทุกครั้งที่แก้ไข — ห้ามนับมือ ดู §10)

**✅ ลำดับความสำคัญ EE-at-3-พระจอมฯ (ตั้ง 12 ก.ค. 2569): เสร็จครบทั้ง 3 วิชาหลักแล้ว** — ผู้ใช้ระบุให้เน้นเนื้อหาที่เกี่ยวข้องกับคณะวิศวกรรมไฟฟ้าของ 3 พระจอมฯ (KMUTT/KMITL/KMUTNB) เป็นพิเศษ ตาม `compareData[]` วิชาที่มีน้ำหนักสูงสุดสำหรับกลุ่มนี้คือ TPAT3 (30-40%) และ A-Level (40-50%, ซึ่งสำหรับสายวิศวกรรมหมายถึง Math1+Physics เป็นหลัก) — **TPAT3, A-Level ฟิสิกส์ และ A-Level คณิต 1 ทั้ง 3 วิชาเจาะลึกจนสมดุลครบทุกหมวดย่อยที่ทำได้แล้ว** (แต่ละหมวด ~8 ข้อเท่ากันหมด) ระหว่างทางพยายามผูกโจทย์เข้ากับ EE โดยตรงเท่าที่ทำได้ (เช่น กฎของโอห์ม, สารกึ่งตัวนำ, ใยแก้วนำแสง, จำนวนเชิงซ้อน↔อิมพีแดนซ์, เกตตรรกะดิจิทัล, Bit Error Rate) — งานต่อจากนี้กลับไปใช้เกณฑ์ % ครอบคลุมเทียบขนาดข้อสอบจริงตามปกติสำหรับ 5 วิชาที่เหลือ (เคมี, คณิต 2, อังกฤษ, TGAT)

**⚠️ อยู่ระหว่างรื้อสร้างเนื้อหาใหม่ทั้งหมด (เริ่ม 12 ก.ค. 2569):** ผู้ใช้ตรวจพบว่าเนื้อหาเดิมทั้งหมดเป็นโจทย์ที่ผมคิดขึ้นเองล้วนๆ ไม่ได้อ้างอิงสไตล์/มาตรฐานจากสถาบันติวหรือข้อสอบจริงเลย แม้จะตรวจสอบ "โครงสร้าง" (จำนวนข้อ/หมวดหมู่) กับ blueprint ทางการแล้วก็ตาม แต่ "สไตล์โจทย์" ไม่เคยถูกตรวจเทียบกับของจริง — กำลังไล่รื้อสร้างใหม่ทีละวิชาโดยวิจัย (WebSearch/WebFetch) ตัวอย่างโจทย์จริงจากสถาบันติวที่มีชื่อเสียงก่อนเขียนทุกครั้ง ดู §6.2 และ §12 สำหรับรายละเอียดและแหล่งอ้างอิงต่อวิชา — **มาตรฐาน 5 ตัวเลือกที่พบตอนรื้อ A-Level ฟิสิกส์ ใช้กับทุกวิชา A-Level ต่อจากนี้ (TPAT3/TGAT ยังเป็น 4 ตัวเลือกตามที่พบจริงจากแหล่งอ้างอิงของแต่ละวิชานั้น)** — **ยืนยันด้วยว่า A-Level อังกฤษใช้ 4 ตัวเลือก (ไม่ใช่ 5 เหมือนฟิสิกส์/เคมี/คณิต) — มาตรฐานตัวเลือกจึงต่างกันไปตามวิชาจริง ไม่ใช่กฎตายตัวทั้งระบบ A-Level ต้องตรวจสอบทีละวิชาเสมอ ห้ามสมมติเอาเอง — เช่นเดียวกับ TGAT: TGAT1/TGAT3 เป็น 4 ตัวเลือก แต่ TGAT2 เป็น 5 ตัวเลือก (ยืนยันจาก smartmathpro โดยตรง)** — ✅ **รื้อสร้างครบทั้ง 8 วิชาแล้ว (12 ก.ค. 2569)** ดู §13 ข้อ 32 สำหรับสรุปภาพรวมทั้งแคมเปญ

| **A-Level คณิต 2** ✅ รื้อสร้างใหม่แล้ว + ครบ 100%+ | 5 | 8 ชุด / 32 ข้อ (107% ของ 30 ข้อจริง) | เนื้อหาเดิมตรงกับหัวข้อจริงอยู่แล้ว (ตรวจสอบแล้วผ่านการเทียบกับ checklist สถาบันติว) — ปรับเป็นตัวเลือก 5 ข้อทั้งหมด ครบทั้ง 2 หมวด — เจาะลึกหมวดเซต/ตรรกศาสตร์/เลขยกกำลัง/ฟังก์ชัน, ลำดับ/ดอกเบี้ย/สถิติ/ความน่าจะเป็น, และจำนวน/พีชคณิตเพิ่ม จนเกิน 100% ของขนาดข้อสอบจริงแล้ว (สมการเลขยกกำลัง, ฟังก์ชันประกอบทั้งสองทิศทาง, ผลต่างเซต/อินเตอร์เซกชัน, กฎเดอมอร์แกน, ผลรวมอนุกรม, ดอกเบี้ยธรรมดา, ฐานนิยม, ลำดับเรขาคณิต, ค่าความจริง AND/OR) |
| **A-Level อังกฤษ** ✅ ตรวจสอบสไตล์แล้ว + ครบทุกหมวดที่ทำได้ | 2 | 18 ชุด / 72 ข้อ (90% ของ 80 ข้อจริง, เกินเพดานที่ทำได้จริง 60 ข้อแล้ว) | **ยืนยันว่าเนื้อหาเดิมใช้รูปแบบถูกต้องอยู่แล้ว (4 ตัวเลือก, dialogue completion) ตรงกับตัวอย่างจริงจาก smartmathpro/webythebrain** — ทักษะการอ่านครบทุกประเภทตาม blueprint แล้ว — เจาะลึกจนครบทุกหมวดย่อยที่ทำได้ทั้ง 9 หมวด (8 ข้อ/หมวด): ไวยากรณ์/คำศัพท์, การอ่านจับใจความ, Text Completion, Paragraph Ordering, Advertisement & Review, News & Article, Graph/Chart Data, Error Identification, Best Headline Selection — **ฟัง-พูด ยังไม่มีเลย** (ต้องมีไฟล์เสียง/TTS ที่สภาพแวดล้อมนี้ไม่มี — ข้อจำกัดถาวรเดียวที่เหลือของวิชานี้) |
| **TGAT** ✅ ตรวจสอบสไตล์แล้ว + ครบทุกหมวดที่ทำได้ | 3 | 45 ชุด / 176 ข้อ (88% ของ 200 ข้อจริง) | **ยืนยันมาตรฐานตัวเลือกแยกตามพาร์ท: TGAT1=4 ตัวเลือก (ถูกอยู่แล้ว), TGAT2=5 ตัวเลือก (แก้จาก 4 เป็น 5 แล้ว), TGAT3=4 ตัวเลือก (ถูกอยู่แล้ว)** — เพิ่มโจทย์แนว "ปริศนาหลายเงื่อนไข" ในหมวดเหตุผล + เจาะลึก TGAT3 จนสมดุลครบทั้ง 4/4 หมวด (11 ข้อเท่ากันหมด) + เจาะลึก TGAT2 Verbal/Numerical/Logical จนถึงชุดที่ 6 ทั้ง 3 หมวด (24 ข้อ/หมวด, 90% ของเพดาน 80 ข้อ) + เจาะลึก **TGAT1 จนครบ 100% ของ 60 ข้อจริงพอดี** ทั้ง 3 หมวด (20 ข้อ/หมวด): บทสนทนาปฏิเสธคำเชิญ/ขอข้อมูลเพิ่ม/เสนอช่วยเหลือ/ปลอบใจ, too...to/reported speech/would rather/present perfect continuous, เรื่องครูสอนภาษาอังกฤษ — TGAT1: 3/3 หมวด **100% พอดี** · TGAT2: 3/4 หมวด 90% ของเพดาน (**มิติสัมพันธ์ = ลิงก์ภายนอก**) — **ข้อจำกัดที่พบแต่แก้ไม่ได้: TGAT3 จริงมีคำถามแบบเลือกได้หลายข้อ/คำถามคู่เชื่อมโยงกัน ซึ่ง UI นี้รองรับแค่ single-choice** (ดู §9) |
| **เสริมความรู้ EE/CpE** | 1 | 1 ชุด / 4 ข้อ | ไม่ใช่สนามสอบจริง (ไม่มีใน topicMap) — ไฟฟ้ากระแสสลับ+ตรรกะดิจิทัล |

**หมายเหตุสำคัญ:** ไม่มีหัวข้อไหนในระบบนี้ที่สถานะเป็น `'ready'` (มีครบ) — ทุกหัวข้อที่ "มี" อยู่ในระดับ `'partial'` เท่านั้น (มีตัวอย่าง 3-4 ข้อต่อหัวข้อย่อย ไม่ใช่ปริมาณเทียบเท่าคลังข้อสอบสถาบันติวจริงที่ควรมี ~150-300 ข้อต่อวิชา — ดูการประเมินนี้ใน §8) — `comingSoon` Set ว่างเปล่าแล้ว (ทุกวิชามีเนื้อหาอย่างน้อยบางส่วน)

## 8. สิ่งที่ยังไม่ได้ทำ / ค้างอยู่ (Outstanding)

เรียงตามลำดับความสำคัญที่ประเมินไว้ระหว่างพัฒนา:

1. ~~มิติสัมพันธ์ (Diagrammatic/Spatial Reasoning)~~ — ✅ **แก้ปัญหาแล้วด้วยวิธี "ลิงก์ออก" แทนการสร้างเอง** (ดู §9 แถวใหม่) — TPAT3 ส่วนที่ 1 (15 ข้อ) และ TGAT2 (20 ข้อ) ยังไม่มีชุดฝึกของเว็บนี้เอง เพราะต้องใช้โจทย์ภาพ/แผนภาพหมุนที่ต้องตรวจสอบด้วยสายตา (ไม่มี screenshot/browser-automation tool ในสภาพแวดล้อมนี้ — ตรวจสอบด้วย ToolSearch แล้วว่าไม่มีจริง ไม่ใช่แค่เดา) แทนที่ด้วย status `'external'` ใหม่ใน `topicMap` ที่ชี้ไปยังลิงก์ภายนอกที่ตรวจสอบแล้ว 4 รายการ (2 จาก mytcas.com blueprint ทางการ + 2 จาก SmartMathPro) ในกลุ่ม `officialLinks` ชื่อ "แหล่งฝึกมิติสัมพันธ์ (ภายนอก)" — **ถ้าจะสร้างชุดฝึกของเว็บนี้เองในอนาคต ยังต้องมีเครื่องมือ visual QA ก่อนอยู่ดี** (ดู §2 หลักการข้อ 4)
2. ~~A-Level คณิตศาสตร์ประยุกต์ 2 ว่างทั้งหมด~~ — ✅ **เติมแล้ว** (5 บทเรียน, 2 ชุด/8 ข้อ ครบทั้ง 2 หมวดตาม blueprint จริง — mytcas.com/blueprint/a-level-62-math2/)
3. ~~A-Level ภาษาอังกฤษ — หมวด "การเขียน" ว่างทั้งหมด~~ — ✅ **เติมแล้ว** (2 ชุด/8 ข้อ: เติมข้อความให้สมบูรณ์ + เรียงลำดับประโยค/ย่อหน้า — ใช้ multiple-choice กับโจทย์เรียงลำดับ ABCD แทนการพิมพ์คำตอบอิสระ ซึ่งเข้ากับ schema `quizzes[]` เดิมโดยไม่ต้องแก้ UI) — **หมวด "ฟัง-พูด" (20 ข้อ) ยังว่างเป็นหมวดเดียวที่เหลือ** เพราะเว็บนี้ไม่มีไฟล์เสียง/TTS จริง (ต่างจากทักษะการพูดของ TGAT1 ที่จำลองด้วยข้อความบทสนทนาล้วนได้ แต่ "ฟัง" ของ A-Level ต้องใช้คลิปเสียงจริงตามรูปแบบข้อสอบ) — ยังคงเป็นข้อจำกัดที่แท้จริง ไม่ใช่แค่ยังไม่ได้ทำ
4. ~~TGAT1 (การสื่อสารภาษาอังกฤษ) ว่างทั้งหมด~~ — ✅ **เติมครบทั้ง 2/2 หมวดแล้ว** หมวด "ทักษะการพูด" (Question-Response + บทสนทนาสั้น, 4 ข้อ) และหมวด "ทักษะการอ่าน" (2 ชุด/8 ข้อ: text completion + reading comprehension — เพิ่งเติม)
5. **ปริมาณเนื้อหาโดยรวมยังห่างจาก "มั่นใจสอบติดได้"** — ประเมินไว้ในบทสนทนาว่าต้องการ **~150-300 ข้อต่อวิชาหลัก** (รวม ~800-1,200 ข้อทั้งระบบ) ถึงจะเทียบเคียงสถาบันติวจริงได้ ตอนนี้มีเฉลี่ย ~59 ข้อต่อวิชา (472 ข้อ/8 วิชา) — **✅ แคมเปญ "รื้อสร้างให้ตรงสไตล์สถาบันติวจริงก่อน" เสร็จครบทั้ง 8 วิชาแล้ว (12 ก.ค. 2569)** — TPAT3, A-Level ฟิสิกส์/เคมี/คณิต1/คณิต2/อังกฤษ, TGAT ทุกวิชาผ่านการวิจัยเทียบกับตัวอย่างข้อสอบจริงจากสถาบันติว (SmartMathPro, AppliedPhysics, webythebrain) แล้วปรับจำนวนตัวเลือก/รูปแบบโจทย์ให้ตรงมาตรฐานจริงต่อวิชา (บางวิชา 4 ตัวเลือก บางวิชา 5 ตัวเลือก บางส่วนของ TGAT2/3 ยังมีรูปแบบที่ UI นี้ทำไม่ได้ — ดู §9) — **งานที่เหลือจากนี้กลับมาเป็น "เพิ่มปริมาณ" ตามปกติ** ให้เข้าใกล้ 150-300 ข้อ/วิชา โดยยึดสไตล์ที่ยืนยันแล้วของแต่ละวิชาเป็นแม่แบบ (ไม่ใช่คิดเองอิสระเหมือนก่อนหน้านี้) — งานนี้เปิดค้างไว้ต่อเนื่องหลาย session
6. ~~โหมดจับเวลาสอบจริง (Mock Exam แบบ timed)~~ — ✅ **เสร็จแล้ว** (สุ่มโจทย์ข้ามวิชา + countdown timer, ดู §5)
7. ~~Export/Import ความคืบหน้าเป็นไฟล์~~ — ✅ **เสร็จแล้ว** (`exportProgress()` / `importProgress()` ในแท็บความคืบหน้า)
8. ~~ข้อมูลกำหนดการ~~ — ✅ **`rounds[]` อัปเดตเป็นปฏิทิน TCAS70 จริงแล้ว** (ค้นคว้า+ตรวจสอบไขว้ ก.ค. 2569) แต่ **`compareData[]` (สัดส่วนคะแนนรายมหาวิทยาลัย) ยังเป็น mock ทั้งหมด** — สาเหตุที่ยังไม่แทนที่: เกณฑ์จริงต่างกันตามคณะ/สาขาในมหาวิทยาลัยเดียวกัน ประกาศเป็น PDF แยกรายโครงการ ไม่ใช่ตารางเดียวที่ดึงมาใช้ได้ตรงๆ (ดู §6.1 สำหรับรายละเอียดกระบวนการตรวจสอบ `rounds[]` และปัญหาที่เจอ)

## 9. ข้อจำกัดที่ทราบอยู่แล้ว (Known limitations — ไม่ใช่บั๊ก เป็นการตัดสินใจที่มีเหตุผล)

|---|---|
| ไม่มีการดึงข้อมูลสดจากเว็บทางการ | เว็บสถาบัน/มหาวิทยาลัยไม่มี public API และหลายเว็บบล็อก iframe (`X-Frame-Options`) — ใช้ลิงก์ตรงแทน (`officialLinks`) |

| localStorage key ยังใช้คำว่า `wongwian` | เศษตกค้างจากตอนโปรเจกต์ชื่อ "วงเวียน" ก่อนเปลี่ยนเป็น "เข็มทิศ" — **ไม่ได้แก้เพราะจะทำให้ข้อมูลผู้ใช้ที่มีอยู่แล้วหายไปโดยไม่จำเป็น** (ถ้าจะแก้ในอนาคต ต้องเขียน migration script ย้ายข้อมูลจาก key เก่าไป key ใหม่ ไม่ใช่แค่ rename ตรงๆ) |
| ไม่มีมิติสัมพันธ์/โจทย์ภาพของตัวเอง | สภาพแวดล้อมพัฒนานี้ไม่มี screenshot/browser-automation tool (ตรวจสอบด้วย ToolSearch แล้ว) จึงเลี่ยงเดาพิกัด SVG โดยไม่ตรวจสอบ — แก้ด้วยการลิงก์ออกไปแหล่งฝึกจริงที่ตรวจสอบแล้วแทน (status `'external'` ใน `topicMap`, ลิงก์อยู่ใน `officialLinks` กลุ่ม "แหล่งฝึกมิติสัมพันธ์") |
| ข้อสอบทุกข้อเป็นเนื้อหาที่เขียนขึ้นใหม่ | ไม่ใช่ข้อสอบเก่าที่หลุดออกมาจริง — ถูกต้องตามหลักวิชาแต่ไม่ควรอ้างว่า "ข้อสอบจริงปีนี้ปีนั้น" (ตั้งแต่ 12 ก.ค. 2569 อ้างอิงสไตล์จากสถาบันติวจริงแล้ว ดู §6.2 — แต่ยังคงเป็นโจทย์ที่เขียนขึ้นใหม่ ไม่ใช่ข้อสอบจริงที่คัดลอกมา) |
| ไม่มีคำถามแบบ "ระบายคำตอบตัวเลข" (numeric entry) | A-Level ทุกวิชามีข้อสอบส่วนหนึ่ง (ปกติ 5/30 ข้อ) เป็นแบบเขียนคำตอบตัวเลขเองแทนการเลือกจากตัวเลือก — UI ของเว็บนี้ (`playQuiz()`) ออกแบบมาสำหรับ multiple-choice (`choices[]`+`correct` index) เท่านั้น จึงจำลองด้วยโจทย์คำนวณแบบปรนัยแทนทั้งหมด (ทดสอบทักษะคำนวณเดียวกัน แต่รูปแบบการตอบต่างจากข้อสอบจริง) |
| ไม่มีคำถามแบบ "เลือกได้หลายข้อ" หรือ "คำถามคู่เชื่อมโยงกัน" (TGAT3) | ข้อสอบ TGAT3 จริงมีบางข้อให้เลือกคำตอบได้มากกว่า 1 ข้อ (ให้คะแนนบางส่วน) และมีคำถามคู่ที่ต้องตอบถูกทั้งคู่จึงได้คะแนน (linked questions) — พบจากตัวอย่างจริงของ smartmathpro (สถานการณ์ปลูกทุเรียนของน้องขวัญ) — UI นี้รองรับแค่ single-choice ต่อคำถามเท่านั้น จึงยังคงใช้รูปแบบเดิม (คำตอบเดียว) ซึ่งทดสอบวิจารณญาณแบบเดียวกันแต่ไม่ตรงรูปแบบข้อสอบจริงเป๊ะๆ |
| ไม่มี backend, ไม่มี auth, ไม่มี multi-user | ตามสเปกเดิม (ต้นแบบเดี่ยว ไม่ใช่ SaaS) |

## 10. วิธีตรวจสอบไฟล์ก่อนส่งมอบ (Verification Workflow)

### 10.0 ก่อนแก้โค้ด: Trace ผลกระทบรอบด้าน (360°) — กฎบังคับ (ตั้ง 12 ก.ค. 2569)

ก่อนแก้โค้ดทุกครั้ง (ไม่ใช่แค่รอบใหญ่) ให้ trace ผลกระทบตามมุมเหล่านี้ก่อนลงมือแก้จริง — ขนาดของการ trace ปรับตามขนาดของการแก้ได้ แต่ห้ามข้ามขั้นนี้ไปเลย:

- **รับ (input):** โค้ดที่จะแก้รับข้อมูล/ถูกเรียกจากไหนบ้าง? ข้อมูลที่รับเข้ามามีรูปร่าง/ข้อสมมติอะไรบ้าง?
- **ส่ง (output):** โค้ดนี้ผลิต/return/เขียน/render อะไรออกไป? ใครเป็นคนอ่านผลลัพธ์นั้นต่อ?
- **การนำไปใช้งาน (usage / call sites):** grep หาทุกจุดที่อ้างอิงถึงฟังก์ชัน/ตัวแปร/id/field ที่กำลังจะแก้ **ทั่วทั้งโปรเจกต์** ไม่ใช่แค่ไฟล์ที่กำลังแก้ — ทุก call site, ทุกที่ที่อ่าน field นั้น, ทุกที่ที่ match id นั้น
- **การประมวลผล (processing/ordering):** มี transformation, side effect (เขียน localStorage, แก้ DOM, global state), หรือ dependency เรื่องลำดับ/เวลา (เช่นลำดับโหลด `<script src>`, ลำดับเรียกฟังก์ชัน) แทรกอยู่ระหว่างทางหรือไม่? มี "สัญญาที่มองไม่เห็น" หรือไม่ (เช่น field ที่ตัว render เรียกตรงๆ โดยไม่มี fallback ถ้าขาด — ดูตัวอย่างใน §4 quizzes[])

ผล trace อาจทำให้พบว่าการแก้ที่ตั้งใจไว้ต้องแก้ "จุดอื่นด้วย" (เช่น เปลี่ยนรูปร่างข้อมูล → ต้องอัปเดตทุกฟังก์ชัน render ที่อ่านข้อมูลนั้น ไม่ใช่แค่ตัวข้อมูลเอง) — ต้องจับให้ครบก่อนเริ่มแก้จริง

### 10.1 หลังแก้โค้ด: Post-deploy Verification

**สภาพแวดล้อมที่ใช้พัฒนาไม่มี browser devtools หรือ screenshot tool** ดังนั้นทุกครั้งที่แก้ `index.html` หรือไฟล์ใน `data/` จะตรวจสอบด้วย Node.js 3 ขั้นตอนนี้เสมอ (แนะนำให้ทำต่อเช่นกัน) — **อัปเดต 12 ก.ค. 2569 ให้ตรงกับโครงสร้างไฟล์แยกใหม่ (ดู §13 ข้อ 19)**

```bash
# 1. แยกสคริปต์หลักจาก index.html (ตัวที่มี const examDates เป็นตัวบ่งชี้ script หลัก
#    เพราะไฟล์มีหลาย <script> tag: CDN, data/*.js placeholder, สคริปต์หลัก) แล้วเช็ค syntax
node -e "
const fs = require('fs');
const html = fs.readFileSync('index.html', 'utf8');
const re = /<script\b[^>]*>([\s\S]*?)<\/script>/g;
let m, target = null;
while ((m = re.exec(html))) { if (m[1].includes('const examDates')) target = m[1]; }
fs.writeFileSync('/path/to/scratch/main.js', target);
"
node --check /path/to/scratch/main.js && echo "MAIN SYNTAX OK"

# 2. เช็ค syntax ของทุกไฟล์ data/*.js แยกตัว (ไฟล์ที่เพิ่ง WriteFile ใหม่/แก้ไข ต้องผ่านก่อนเสมอ)
for f in data/*.js; do node --check "$f" && echo "OK: $f"; done

# 3. รันจริงแบบจำลองลำดับโหลดของเบราว์เซอร์ทุกไฟล์ (data/*.js ตามลำดับใน index.html ก่อน
#    แล้วค่อยสคริปต์หลัก) ด้วย DOM มินิม็อค พร้อม inject window.__DUMP เพื่อดึงค่านับที่แน่นอน
#    (ห้ามนับมือ — โปรเจกต์นี้เคยนับผิดมาแล้วจริง)
node -e "
const fs = require('fs');
function makeEl(){ /* ...mock element with innerHTML/textContent/classList/etc... */ }
global.window = global;
global.document = { getElementById(){return makeEl();}, querySelector(){return makeEl();},
  querySelectorAll(){return [];}, createElement(){return makeEl();}, addEventListener(){},
  body: makeEl(), documentElement: makeEl() };
global.localStorage = { getItem(){return null;}, setItem(){}, removeItem(){} };
global.matchMedia = () => ({ matches:false, addEventListener(){}, addListener(){} });
global.location = { search:'', href:'file:///index.html', hash:'' };

const order = ['tpat3','alevel-math1','alevel-phys','alevel-chem','alevel-math2','alevel-eng','tgat','enrichment'];
for (const s of order) eval(fs.readFileSync('data/' + s + '.js', 'utf8'));

const mainScript = fs.readFileSync('/path/to/scratch/main.js', 'utf8');
const patched = mainScript.replace('function loadState()', 'window.__DUMP = { lessons, quizzes }; function loadState()');
eval(patched);
const d = global.__DUMP;
console.log('lessons:', d.lessons.length, 'quizzes:', d.quizzes.length, 'questions:', d.quizzes.reduce((a,q)=>a+q.questions.length,0));
const qIds = d.quizzes.map(q=>q.id), lIds = d.lessons.map(l=>l.id);
console.log('duplicate quiz ids:', qIds.filter((id,i)=>qIds.indexOf(id)!==i));
console.log('duplicate lesson ids:', lIds.filter((id,i)=>lIds.indexOf(id)!==i));
"
```

ขั้นตอน 3 นี้สำคัญกว่าตอนไฟล์เดียว เพราะตอนนี้ต้องตรวจสอบด้วยว่า **ลำดับการโหลดไฟล์ตรงกับที่ประกาศจริงใน `index.html`** (data files ต้องโหลดก่อนสคริปต์หลักเสมอ ไม่งั้น `window.KEMTID_DATA` จะยังว่างตอนสคริปต์หลักอ่านค่า)

หลังผ่านทั้ง 3 ขั้น ค่อยเปิดไฟล์จริงในเบราว์เซอร์ (`Start-Process index.html` บน PowerShell) เพื่อให้ผู้ใช้ตรวจสอบด้วยตาอีกชั้น — **การตรวจ 3 ขั้นข้างต้นจับ error ทางโค้ดได้ แต่จับปัญหา visual/UX ไม่ได้** ต้องอาศัยคนดูจริงเสมอสำหรับเรื่องหน้าตา

**อย่าหยุดแค่ "รันผ่านไม่ error"** — ให้ย้อนกลับไปเช็คจุดที่ระบุไว้ตอน trace (§10.0) ทีละจุดว่ายังทำงานถูกต้องจริงตามที่คาดหวัง ไม่ใช่แค่ไม่ crash เช่น ถ้า trace บอกว่าฟังก์ชัน A ถูกเรียกใช้จาก 3 จุด ต้องยืนยันว่าทั้ง 3 จุดยังได้ผลลัพธ์ที่ถูกต้อง ไม่ใช่แค่จุดที่แก้ไขโดยตรง

## 11. แนวทางสำหรับพัฒนาต่อ (Conventions ที่ต้องทำตาม)

### เพิ่มชุดข้อสอบใหม่
1. เปิดไฟล์ **`data/{subjectId}.js` ของวิชานั้นโดยตรง** (เช่นเพิ่มข้อสอบฟิสิกส์ → แก้ `data/alevel-phys.js`) — **ไม่ต้องแตะ `index.html`**
2. เพิ่ม object เข้าไปใน `window.KEMTID_DATA.quizzes.push(...)` ที่มีอยู่แล้วในไฟล์นั้น ให้ครบทุก field ตามโครงสร้างใน §4 (`intro`, `hint`, `steps`, `tip` — ห้ามขาด)
3. ตั้ง `id` ตามแพทเทิร์น `q_{subjectId แบบย่อ}_{ลำดับ หรือ หัวข้อสั้นๆ}` เช่น `q_alevel_phys_4`, `q_tpat3_spatial` — **id ต้องไม่ซ้ำกับที่มีอยู่ทั้งระบบ ไม่ใช่แค่ในไฟล์เดียวกัน** (ตรวจด้วยสคริปต์ §10 ขั้นที่ 3 เสมอ เพราะข้าม subjectId ไม่ได้เตือนกันเอง)
4. `subjectId` ต้องตรงกับ id ใน `subjects[]` เป๊ะๆ **และต้องตรงกับชื่อไฟล์ที่กำลังแก้ด้วย** (ป้องกันเผลอใส่ข้อสอบผิดวิชาไปอยู่ผิดไฟล์)
5. **ตรวจคำตอบด้วยมือทุกข้อ** — โปรเจกต์นี้เขียนโจทย์เองทั้งหมด ไม่ได้ copy จากแหล่งที่ verify แล้ว ต้องคำนวณ/ตรวจสอบความถูกต้องเชิงวิชาการเองก่อนใส่
6. อัปเดต `topicMap` ใน `index.html` ให้ status ตรงความจริง (`none`→`partial` เมื่อเริ่มมีเนื้อหา)
7. รันตรวจสอบตาม §10 ก่อนส่งมอบทุกครั้ง (ต้องรันไฟล์ data ทั้งหมดตามลำดับ ไม่ใช่แค่ไฟล์ที่เพิ่งแก้)

### เพิ่มบทเรียนใหม่
- แก้ที่ `data/{subjectId}.js` ของวิชานั้น เพิ่มเข้า `window.KEMTID_DATA.lessons.push(...)` — โครงสร้างดู §4 (`formula: '—'` ถ้าไม่มีสูตร)
- ใช้ id ที่ไม่ซ้ำกับที่มีอยู่ทั้งระบบ (เช็คด้วย grep `id: 'l` ในทุกไฟล์ `data/*.js` ก่อนตั้ง id ใหม่)

### เพิ่มวิชาใหม่ทั้งหมด
1. สร้างไฟล์ใหม่ `data/{subjectId}.js` ตามรูปแบบไฟล์อื่นในโฟลเดอร์เดียวกัน (มี `window.KEMTID_DATA = window.KEMTID_DATA || { lessons: [], quizzes: [] };` แล้ว push เข้าไป)
2. เพิ่ม `<script src="data/{subjectId}.js"></script>` ใน `index.html` ในกลุ่ม `<script src="data/...">` ทั้ง 8 บรรทัด (ต้องอยู่ก่อนสคริปต์หลักเสมอ)
3. เพิ่มใน `subjects[]` (ใน `index.html`)
4. ถ้ายังไม่มีเนื้อหา ใส่ไว้ใน `comingSoon` Set ก่อน (เอาออกเมื่อมีเนื้อหาจริงแล้ว)
5. เพิ่ม entry ใน `topicMap` **ต้องตรวจสอบกับ Test Blueprint จริงจาก mytcas.com ก่อนเสมอ** (ดู §6 บทเรียนที่ได้มา) ห้ามเดาโครงสร้างข้อสอบเอง

### เพิ่มลิงก์ทางการใหม่
- **ต้อง WebSearch ตรวจสอบ URL จริงก่อนใส่ทุกครั้ง** ห้ามเดา/แต่ง URL จากความจำ (ดูหลักการ §2 ข้อ 3)

### ทั่วไป
- **โครงสร้างไฟล์ (12 ก.ค. 2569 เป็นต้นไป):** `index.html` = โครง+ฟังก์ชัน, `data/*.js` = เนื้อหารายวิชา — แยกไฟล์โดยตั้งใจแล้ว (เดิมเป็นไฟล์เดียว ดู §13 ข้อ 19 สำหรับเหตุผลที่เปลี่ยน) **ผู้ใช้แจกจ่ายทั้งโฟลเดอร์เป็น zip เสมอ ไม่ใช่ไฟล์เดียว** — อย่าย้อนกลับไปยัดข้อมูลกลับเข้า `index.html` โดยไม่ปรึกษาก่อน
- ภาษาที่ใช้ในเนื้อหา UI/ข้อสอบ: **ภาษาไทยทั้งหมด** (ยกเว้นศัพท์เทคนิคเฉพาะทาง เช่น ชื่อสนามสอบ TGAT/TPAT/A-Level, สูตรคณิต/ฟิสิกส์)

20. **ตั้งกฎบังคับ: Trace ผลกระทบ 360° ก่อนแก้โค้ด + Post-deploy verification หลังแก้ทุกครั้ง** — ผู้ใช้สั่งตั้งกฎนี้ (12 ก.ค. 2569) ต่อจากรอบ refactor แยกไฟล์ (ข้อ 19) เพิ่ม §10.0 (ขั้นตอน trace: รับ/ส่ง/การนำไปใช้งาน/การประมวลผล ก่อนแก้) และขยาย §10.1 ให้ย้ำว่าหลังตรวจสอบด้วย Node ต้องย้อนกลับไปเช็คจุดที่ trace ไว้ทีละจุดด้วย ไม่ใช่แค่ "รันผ่านไม่ error" — ปรับ §2 หลักการข้อ 6 ให้ตรงกัน
21. เจาะลึก A-Level เคมีตามลำดับที่วางไว้ในข้อ 18 (วิชาแรกหลังปรับสถาปัตยกรรมไฟล์แยก — เป็นการทดสอบ workflow ใหม่ไปในตัว) — ใช้กฎ 360° trace จากข้อ 20 ก่อนแก้: ระบุว่ามีความเสี่ยงหลักคือ id ชนกันข้ามไฟล์และ topicMap ต้องตรงความจริง — เพิ่ม 4 ชุดใหม่ (16 ข้อ) เจาะ 4 หัวข้อที่ไม่เคยมีเนื้อหาเลย (เคมีอินทรีย์/พอลิเมอร์, แก๊ส/ปริมาณสัมพันธ์, อัตราการเกิดปฏิกิริยา/สมดุลเคมี, เคมีไฟฟ้า) A-Level เคมีขยับจาก 8→24 ข้อ (~69% ของ 35 ข้อจริง) — post-deploy verification ยืนยัน 31 บทเรียน/37 ชุด/142 ข้อ ไม่มี id ซ้ำ ทุกคำถามมี field required (hint/steps/tip) ครบ
22. เจาะลึก A-Level คณิต 1 และคณิต 2 พร้อมกันตามคำขอผู้ใช้ — คณิต 1 เพิ่ม 4 ชุดใหม่ (16 ข้อ) เจาะหัวข้อที่ไม่เคยมี/บางมาก (สถิติ: มัธยฐาน/ฐานนิยม/พิสัย ซึ่งเดิมไม่มีเลยมีแต่ความน่าจะเป็น, ลิมิตของฟังก์ชัน, เอกลักษณ์ตรีโกณและมุมพิเศษเพิ่มเติม, อนุกรมเรขาคณิต/ผลรวม) ขยับจาก 12→28 ข้อ — คณิต 2 เพิ่ม 3 ชุดใหม่ (12 ข้อ) เจาะลำดับเลขคณิต/ฟังก์ชันประกอบ, มัธยฐาน/ฐานนิยม/พิสัย, หลักการนับ (multiplication principle) ขยับจาก 8→20 ข้อ — post-deploy verification ยืนยัน 31 บทเรียน/44 ชุด/170 ข้อ ไม่มี id ซ้ำ ทุกคำถามมี field required ครบและ `correct` index อยู่ในช่วงที่ถูกต้องทุกข้อ
23. ผู้ใช้ให้อิสระเลือกวิชาต่อเอง โดยยึดหลัก "ครอบคลุมและมีประโยชน์มากที่สุด" — เลือกปิดช่องว่างที่สถานะยังเป็น `'none'` (ไม่มีเนื้อหาเลย) ก่อนเพิ่มความลึกในหมวดที่มีอยู่แล้ว เพราะให้คุณค่าส่วนเพิ่มสูงกว่า พบ 2 จุดที่ปิดได้จริงโดยไม่ต้องมีไฟล์เสียง: **TGAT1 ทักษะการอ่าน** (เพิ่ม 2 ชุด/8 ข้อ: text completion + reading comprehension, 0→8 ข้อ) และ **A-Level อังกฤษ ทักษะการเขียน** (เพิ่ม 2 ชุด/8 ข้อ: text completion + เรียงลำดับประโยค/ย่อหน้าแบบ multiple-choice ABCD, 0→8 ข้อ) อัปเดต `topicMap` ทั้งสองจุดจาก `'none'`→`'partial'` ให้ตรงความจริงตามหลักการ §2 ข้อ 5 — post-deploy verification ยืนยัน 31 บทเรียน/48 ชุด/186 ข้อ ไม่มี id ซ้ำ — **หลังรอบนี้ทุกหมวดในทุกวิชามีเนื้อหาอย่างน้อยบางส่วนแล้ว ยกเว้น 2 จุดที่มีข้อจำกัดทางเทคนิคจริง** (มิติสัมพันธ์ต้องใช้ภาพ, ฟัง-พูด A-Level อังกฤษต้องใช้เสียง) งานที่เหลือจากนี้คือเพิ่มความลึกล้วนๆ ไม่ใช่ปิดช่องว่างเชิงหมวดอีกต่อไป
24. เจาะลึก TGAT ต่อตามที่ผู้ใช้เลือก (จากตัวเลือกที่เสนอไว้ในข้อ 23) — เลือกเจาะ TGAT2 ทั้ง 3 หมวดที่ทำได้โดยไม่ต้องใช้ภาพ (Verbal/Numerical/Logical Reasoning) เป็นหลัก ด้วยเหตุผลว่าเป็นทักษะที่ฝึกจากรูปแบบซ้ำๆ ได้ประโยชน์จากปริมาณโจทย์มากกว่า TGAT3 ที่เป็นโจทย์เชิงวิจารณญาณซึ่งมีรูปแบบจำกัดกว่า — เพิ่มชุดที่ 2 ให้ทั้ง 3 หมวด (Verbal: คำพ้อง/ตรงข้าม/หมวดหมู่/อุปมา, Numerical: ลำดับตัวเลข/เปอร์เซ็นต์/อัตราส่วนงาน/ค่าเฉลี่ย, Logical: syllogism/modus tollens/transitive/disjunctive) + เพิ่ม TGAT1 Question-Response ชุดที่ 2 (หมวดที่ยังบางที่สุดในกลุ่ม TGAT1) รวม 4 ชุดใหม่ (16 ข้อ) TGAT ขยับจาก 36→52 ข้อ — post-deploy verification ยืนยัน 31 บทเรียน/52 ชุด/202 ข้อ ไม่มี id ซ้ำ ทุก field ครบ
25. เจาะลึก A-Level อังกฤษต่อตามคำขอผู้ใช้ — trace ก่อนแก้พบว่าหมวด "ทักษะการอ่าน" (40 ข้อจริง สัดส่วนใหญ่ที่สุดของข้อสอบ) ยังมีแค่แกรมมาร์พื้นฐาน+เรื่องเล่าสั้นๆ ทั้งที่ blueprint ระบุประเภทข้อความไว้ชัดเจน 5 แบบ (โฆษณา รีวิว ข่าว กราฟ/แผนภูมิ บทความ) — เพิ่ม 3 ชุดใหม่ครอบคลุมทุกประเภทที่เหลือ (โฆษณา+รีวิว, ข่าว+บทความ, กราฟ/แผนภูมิ — ใช้ข้อมูลตัวเลขบรรยายเป็นข้อความแทนภาพจริงเพื่อคงทักษะการอ่าน-ตีความข้อมูลแบบเดียวกัน) และเพิ่ม 1 ชุดทักษะการเขียนแบบหาข้อผิดพลาดในประโยค (error identification) เพื่อความสมดุล A-Level อังกฤษขยับจาก 16→32 ข้อ (~40% ของ 80 ข้อจริง) — post-deploy verification ยืนยัน 31 บทเรียน/56 ชุด/218 ข้อ ไม่มี id ซ้ำ ทุก field ครบ
26. ผู้ใช้ให้เลือกวิชาต่อเองอีกครั้ง — เลือก TGAT3 เพราะเป็นวิชาที่บางที่สุดเทียบกับขนาดจริงตอนนั้น (12/60 ข้อ = 20%) และ TGAT ใช้ยื่นแทบทุกรอบทุกคณะ — เพิ่มชุดที่ 2 ให้ครบทั้ง 4 หมวด (Value Creation, Complex Problem Solving, Emotional Management, Active Citizenship) ออกแบบสถานการณ์ใหม่ไม่ซ้ำของเดิม (เช่น root cause analysis, trade-off ระหว่างซัพพลายเออร์, feedback แบบเน้นพฤติกรรม, เวทีรับฟังความคิดเห็นสาธารณะ) รวม 4 ชุดใหม่ (16 ข้อ) TGAT ขยับจาก 52→68 ข้อ (~34% ของ 200 ข้อจริง) — post-deploy verification ยืนยัน 31 บทเรียน/60 ชุด/234 ข้อ ไม่มี id ซ้ำ ทุก field ครบ
27. **จุดเปลี่ยนสำคัญ: ผู้ใช้ตั้งคำถามว่าเนื้อหาข้อสอบทั้งหมดอ้างอิงสถาบันติว/ติวเตอร์ที่ประสบความสำเร็จจริงหรือไม่** — ตรวจสอบแล้วพบว่า **ไม่ใช่**: ตรวจสอบมาตลอดแค่ "โครงสร้าง" (จำนวนข้อ/หมวดหมู่ตาม blueprint) แต่ "สไตล์โจทย์จริง" ไม่เคยเทียบกับแหล่งภายนอกเลย (ดู §6.2 บทเรียนเต็ม) — ผู้ใช้สั่งให้ "รื้อทำใหม่ทั้งหมดโดยอ้างอิงสถาบันติวเป็นหลัก" เริ่มจาก **TPAT3** วิจัยผ่าน WebSearch/WebFetch เทียบกับ SmartMathPro (ตัวอย่างโจทย์ 11 ข้อ verbatim) และ AppliedPhysics พบความไม่ตรงที่ชัดเจน 2 จุด: (1) ด้านตัวเลขของจริงมีโจทย์แนว "นิยามการดำเนินการเอง" (a@b=2a+b แบบตัวอย่างจริงที่ยืนยันคำตอบแล้ว) เป็นสัดส่วนสำคัญ ซึ่งเนื้อหาเดิมไม่มีเลย (2) หมวด "ความสนใจข่าวสาร" ของจริงต้องการความรู้ข่าวเฉพาะช่วงเวลา (เช่นกรณีฟุกุชิมะ) ซึ่งรักษาความทันสมัยต่อเนื่องไม่ได้ — รื้อสร้าง TPAT3 ใหม่ทั้ง 8 ชุด (32 ข้อ): เพิ่มโจทย์นิยามการดำเนินการเอง, เพิ่มโจทย์ความรู้วงจรไฟฟ้า/สมบัติสสารในหมวดกลศาสตร์ (ตามตัวอย่างจริงที่พบ), ปรับความคิดเชิงวิทย์เป็นแนว "ทำนายผลการทดลอง" (เช่น เจาะรูขวดน้ำที่ระดับต่างกัน — ตามตัวอย่างจริง), แทนที่หมวดข่าวสารด้วยเหตุการณ์ประวัติศาสตร์วิทยาศาสตร์ที่ไม่ล้าสมัย (Apollo 11, เพนิซิลลิน, จีโนมมนุษย์, สัมพัทธภาพ) แทนข่าวปัจจุบันที่รักษาไม่ได้ — post-deploy verification ยืนยัน 31 บทเรียน/60 ชุด/236 ข้อ ไม่มี id ซ้ำ ทุก field ครบ — **งานนี้ยังไม่จบ ต้องทำแบบเดียวกันกับอีก 7 วิชาที่เหลือ (A-Level ฟิสิกส์/เคมี/คณิต1/คณิต2/อังกฤษ, TGAT) ในรอบถัดไป**
28. รื้อสร้าง **A-Level ฟิสิกส์** ตามแผนในข้อ 27 — วิจัยเทียบกับ `smartmathpro.com/article/a-level-physics/` (ตัวอย่างโจทย์ 3 ข้อ verbatim) และ `webythebrain.com` พบมาตรฐานสำคัญที่กระทบทุกวิชา A-Level ไม่ใช่แค่ฟิสิกส์: **ข้อสอบจริงใช้ตัวเลือก 5 ข้อ ไม่ใช่ 4** และมีโจทย์แบบ "พิจารณาข้อความ ก./ข./ค. แล้วเลือกว่าข้อใดถูก" ที่ไม่เคยมีมาก่อน — รื้อสร้างทั้ง 5 หมวด (กลศาสตร์, คลื่นกล-แสง, ไฟฟ้า-แม่เหล็ก, อุณหพลศาสตร์, ฟิสิกส์แผนใหม่) เป็น 5 ชุด/20 ข้อ ตัวเลือก 5 ข้อทั้งหมด มีโจทย์แบบ ก./ข./ค. อย่างน้อย 1 ข้อ/หมวด และปรับโจทย์คำนวณให้สมจริงขึ้น (เช่น โพรเจกไทล์แบบหาความสูงคงเหลือที่เวลา t ตามตัวอย่างจริง) — พบด้วยว่าข้อสอบจริง 5/30 ข้อเป็นแบบระบายคำตอบตัวเลข ซึ่ง UI เว็บนี้รองรับไม่ได้ (บันทึกเป็นข้อจำกัดถาวรใน §9) — post-deploy verification ยืนยัน 31 บทเรียน/58 ชุด/228 ข้อ (ลดลงจากเดิมเพราะเน้นคุณภาพ/ความสมจริงมากกว่าปริมาณ) ไม่มี id ซ้ำ ทุกคำถามมี 5 ตัวเลือกครบ — **มาตรฐาน 5 ตัวเลือกนี้จะใช้กับทุกวิชา A-Level ที่รื้อสร้างต่อจากนี้ (ไม่ใช้กับ TPAT3/TGAT ซึ่งยืนยันแล้วว่าเป็น 4 ตัวเลือกจริง)**
29. รื้อสร้าง **A-Level เคมี** ตามแผนในข้อ 27 — วิจัยเทียบกับ `webythebrain.com/article/chemistry-a-level-test` และ `chemistryk.com` แต่ **ทั้งสองแหล่งแสดงโจทย์เป็นรูปภาพ ดึงข้อความ verbatim ไม่ได้เลย** (ต่างจาก TPAT3/ฟิสิกส์ที่ได้ตัวอย่างข้อความจริง) — ตัดสินใจใช้มาตรฐาน 5 ตัวเลือกที่ยืนยันแล้วจากฟิสิกส์แทน (เพราะเป็นมาตรฐานระดับ A-Level ทั้งระบบ ไม่ใช่เฉพาะวิชา) แต่บันทึกไว้ชัดเจนว่าไม่ได้ยืนยันสไตล์เฉพาะของเคมีโดยตรง — แปลง 6 ชุดเดิมเป็น 5 ตัวเลือกทั้งหมด (24 ข้อ) และเพิ่มโจทย์แบบพิจารณาข้อความ ก./ข./ค. 3 ข้อ (พันธะเคมี, ไฮโดรคาร์บอน, สมดุลเคมีแบบผสม 3 ปัจจัยในข้อเดียว) แทนที่โจทย์เดิมที่อ่อนที่สุดในแต่ละชุด — post-deploy verification ยืนยัน 31 บทเรียน/58 ชุด/228 ข้อ (ไม่เปลี่ยนจากรอบฟิสิกส์เพราะเคมีคงจำนวนชุด/ข้อเดิมไว้ แค่ปรับรูปแบบ) ไม่มี id ซ้ำ ทุกคำถามมี 5 ตัวเลือกครบ
30. รื้อสร้าง **A-Level คณิต 1 และคณิต 2** พร้อมกัน — วิจัยเทียบกับ `smartmathpro.com/article/checklist-a-level/` (รายชื่อบทพร้อมน้ำหนักคะแนนแยกคณิต 1/2 ของสถาบันติวจริง) พบ **ช่องว่างเชิงโครงสร้างที่สำคัญที่สุดเท่าที่เจอมาในรอบรื้อสร้างนี้**: เซตและตรรกศาสตร์เป็น "บทต้องเก็บ" (น้ำหนักคะแนนสูงสุดบทหนึ่ง) ในคณิต 1 ด้วยเช่นกัน ไม่ใช่แค่คณิต 2 อย่างที่เข้าใจไว้เดิม — เนื้อหาคณิต 1 เดิมไม่มีหมวดนี้เลยแม้แต่ข้อเดียว — เพิ่มชุดใหม่ "เซตและตรรกศาสตร์" + "ฟังก์ชันเอกซ์โพเนนเชียลและลอการิทึม" (อีกบทน้ำหนักสูงที่ขาดไป) ปรับ 4 ชุดเดิมเป็น 5 ตัวเลือก ลดความสำคัญตรีโกณ/ภาคตัดกรวยลงตามคำแนะนำสถาบันติวจริง (ยืนยันว่าเป็นบทน้ำหนักน้อยที่สุด ไม่ใช่แค่เดา) อัปเดต `topicMap` เพิ่มหมวดที่ขาดไปด้วย คณิต 1 ขยับจาก 12 หมวดครอบคลุมไม่ครบ→7/7 หมวดครบ (28→24 ข้อ, ลดจำนวนแต่ครอบคลุมโครงสร้างถูกต้องกว่าเดิม) — คณิต 2 ตรวจสอบแล้วพบว่าเนื้อหาเดิมตรงกับหัวข้อจริงอยู่แล้ว ไม่ต้องปรับโครงสร้าง แค่แปลง 5 ชุดเดิมเป็น 5 ตัวเลือก (20 ข้อ) — post-deploy verification ยืนยัน 32 บทเรียน/57 ชุด/224 ข้อ ไม่มี id ซ้ำ ทุกคำถามมี 5 ตัวเลือกครบ
31. รื้อสร้าง **A-Level อังกฤษ** ตามแผนในข้อ 27 — วิจัยเทียบกับ `smartmathpro.com/article/a-level-eng/` (ได้ตัวอย่างโจทย์ verbatim ครบทั้ง 3 หมวด) และ `webythebrain.com` **พบว่า A-Level อังกฤษใช้ 4 ตัวเลือก ไม่ใช่ 5 เหมือนฟิสิกส์/เคมี/คณิต — ยืนยันว่ามาตรฐานจำนวนตัวเลือกต่างกันไปตามวิชาจริง ไม่ใช่กฎตายตัวทั้งระบบ A-Level ต้องตรวจสอบทีละวิชาเสมอ** ตรวจสอบเนื้อหาเดิม (dialogue completion แบบ Question-Response, reading comprehension แบบสกัดข้อเท็จจริง) แล้วพบว่า **ตรงกับสไตล์จริงอยู่แล้ว ไม่ต้องรื้อสร้างใหม่ทั้งหมดเหมือนวิชาอื่น** — พบแนวโจทย์ใหม่ที่ไม่เคยมี: "เลือกพาดหัวข่าวที่ดีที่สุด" (best headline) จากตัวอย่างข่าวเครื่องบินไร้คนขับจริง — เพิ่มชุดใหม่ 1 ชุด/4 ข้อ ฝึกแยกหัวข้อที่แคบ/กว้างเกินไป/พูดเกินจริง A-Level อังกฤษขยับจาก 32→36 ข้อ (~45% ของ 80 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/58 ชุด/228 ข้อ ไม่มี id ซ้ำ ทุกคำถามมี 4 ตัวเลือกครบ (ยืนยันว่ายังคงเป็นมาตรฐานถูกต้องสำหรับวิชานี้)
32. รื้อสร้าง **TGAT** (วิชาสุดท้ายของแคมเปญ 8 วิชา) — วิจัยเทียบกับ `smartmathpro.com/article/tgat2-reasoning/`, `tgat2-languages-skill/` และ `tgat3-value-creation/` พบว่า **มาตรฐานจำนวนตัวเลือกต่างกันแม้อยู่ในวิชา TGAT เดียวกัน**: TGAT1=4 ตัวเลือก (ถูกอยู่แล้ว), TGAT2=5 ตัวเลือก (ยืนยันจากข้อความ "ข้อสอบเป็นแบบปรนัย (5 ตัวเลือก)" ตรงๆ — เดิมเป็น 4 ผิดมาตรฐาน แก้แล้วทั้ง 24 ข้อ), TGAT3=4 ตัวเลือก (ถูกอยู่แล้ว) — พบด้วยว่า TGAT2 ด้านเหตุผลจริงมีโจทย์ปริศนาหลายเงื่อนไข (จัดตำแหน่ง/จับคู่ ต้องใช้ตารางช่วยคิด) ซับซ้อนกว่า syllogism 2 ข้อสมมติที่มีอยู่เดิม — เพิ่มชุดใหม่ "ปริศนาหลายเงื่อนไข" 4 ข้อจำลองสไตล์นี้แบบ single-choice — พบว่า TGAT3 จริงมีคำถามเลือกได้หลายข้อ/คำถามคู่เชื่อมโยงกันจากสถานการณ์เดียว (เช่นตัวอย่างปลูกทุเรียนจริง) ซึ่ง UI นี้ไม่รองรับ บันทึกเป็นข้อจำกัดถาวรใน §9 — TGAT ขยับจาก 68→72 ข้อ (18→19 ชุด) — post-deploy verification ยืนยัน 32 บทเรียน/59 ชุด/232 ข้อ ไม่มี id ซ้ำ ทุก field ครบ **✅ รื้อสร้างครบทั้ง 8 วิชาแล้ว — จบแคมเปญ "ตรวจสอบสไตล์กับสถาบันติวจริง" ที่เริ่มจากคำถามของผู้ใช้ในข้อ 27**
33. ผู้ใช้ให้เลือกวิชาต่อเองอีกครั้งหลังจบแคมเปญตรวจสอบสไตล์ — เลือก **TGAT1** เพราะเมื่อคิดเป็นเปอร์เซ็นต์ของขนาดข้อสอบจริงแล้ว TGAT (~36% ของ 200 ข้อ) บางที่สุดในบรรดา 8 วิชา และภายใน TGAT เอง TGAT1 บางที่สุด (16/60 = ~27%) ทั้งที่ TGAT ใช้ยื่นแทบทุกรอบทุกคณะ — เพิ่ม Question-Response ชุดที่ 3 (4 ข้อ: แสดงความยินดี, ขอร้องสุภาพ, ถามประสบการณ์, บทสนทนาต่อเนื่องเรื่องเวลา) และการอ่านชุดที่ 2 ทั้ง text completion (past perfect, look forward to + gerund, if-conditional, apologize for) และ reading comprehension (เรื่องบรรณารักษ์ที่เชียงใหม่) รวม 4 ชุดใหม่ (16 ข้อ) TGAT1 ขยับจาก 16→28 ข้อ (~47% ของ 60 ข้อจริง) TGAT รวมขยับจาก 72→84 ข้อ — post-deploy verification ยืนยัน 32 บทเรียน/62 ชุด/244 ข้อ ไม่มี id ซ้ำ ทุก field ครบ
34. ผู้ใช้ให้เลือกวิชาต่อเองอีกครั้ง — พบว่า TGAT1/2/3 สมดุลกันแล้วที่ 28 ข้อเท่ากันหมด จึงมองข้ามระดับ TGAT ไปเทียบทั้ง 8 วิชา พบว่า TPAT3 (46%) และ A-Level อังกฤษ (45%) ต่ำสุด — เลือก **TPAT3** เพราะมี "เพดานที่ทำได้จริง" สูงกว่าอังกฤษ (อังกฤษมี 20/80 ข้อถูกบล็อกถาวรจากข้อจำกัดเรื่องเสียง) และมีน้ำหนักคะแนนกระจุกตัวสูงสุดในบรรดาวิชาทั้งหมด — เพิ่ม 3 ชุดใหม่ (12 ข้อ) เจาะ 3 หมวดที่เหลือบางที่สุด (~53% เท่ากัน ไม่นับความสนใจข่าวที่ 80% แล้ว): ด้านตัวเลข (ลำดับอัตราส่วนคงที่, นิยามการดำเนินการเองแบบใหม่, แบ่งเงินตามอัตราส่วน 3 ฝ่าย), ด้านกลศาสตร์ (โมเมนต์, ความดัน P=F/A, รอกผสม, จุดศูนย์ถ่วง), ความคิดเชิงวิทยาศาสตร์ (ตัวแปรต้น-ตาม-ควบคุมในการทดลอง, user testing, แรงเสียดทานกับระยะเบรก) TPAT3 ขยับจาก 32→44 ข้อ (~63% ของ 70 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/65 ชุด/256 ข้อ ไม่มี id ซ้ำ ทุก field ครบ
35. ผู้ใช้ให้เลือกวิชาต่อเองอีกครั้ง — คำนวณเปอร์เซ็นต์ครอบคลุมของทั้ง 8 วิชาใหม่ พบว่า **TGAT ต่ำสุด (42%)** และเมื่อคิดเทียบกับ "เพดานที่ทำได้จริง" (หักส่วนที่บล็อกถาวรออก) TGAT ยังอยู่ที่ 47% ของส่วนที่ทำได้ ต่ำกว่าอังกฤษที่ 60% — เลือก **TGAT** ต่อ ภายใน TGAT พบว่า TGAT1/2/3 สมดุลกันแล้วที่ 28 ข้อเท่ากัน แต่ภายใน TGAT2 เอง Verbal และ Numerical ยังบางกว่า Logical (8/20=40% เทียบกับ Logical 12/20=60%) — เพิ่มชุดที่ 3 ให้ทั้ง Verbal (ความสัมพันธ์แบบส่วนประกอบ-สิ่งทั้งหมด, คำศัพท์ใหม่) และ Numerical (โจทย์อายุ, โจทย์ผสมสารละลาย, ความน่าจะเป็นจากตัวหาร, ลำดับกำลังสอง) รวม 2 ชุดใหม่ (8 ข้อ) TGAT2 Verbal/Numerical ขยับจาก 8→12 ข้อ เท่ากับ Logical แล้ว TGAT รวมขยับจาก 84→92 ข้อ (~46% ของ 200 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/67 ชุด/264 ข้อ ไม่มี id ซ้ำ ทุก field ครบ
36. **ผู้ใช้ตั้งเกณฑ์ความสำคัญใหม่: ให้เน้นเนื้อหาที่เกี่ยวข้องกับคณะวิศวกรรมไฟฟ้าของ 3 พระจอมฯ เป็นพิเศษ** — พิจารณาจาก `compareData[]` พบว่า TPAT3 (30-40%) และ A-Level Math1+Physics (รวมอยู่ในน้ำหนัก 40-50%) เป็นวิชาหลักของกลุ่มนี้ — เลือกเจาะ **A-Level ฟิสิกส์ หมวดไฟฟ้าและแม่เหล็ก** ก่อน เพราะเป็นหมวดที่ตรงกับ "วิศวกรรมไฟฟ้า" โดยตรงที่สุดในบรรดาทุกวิชา/หมวดที่มี เพิ่มชุดใหม่ 1 ชุด (4 ข้อ) ครอบคลุมแนวคิดที่เป็นรากฐานของวิศวกรรมไฟฟ้าโดยตรง: กำลังไฟฟ้า (P=VI, P=I²R), วงจรขนาน, การเหนี่ยวนำแม่เหล็กไฟฟ้า (หลักการเครื่องกำเนิดไฟฟ้า/หม้อแปลง), แรงบนลวดตัวนำในสนามแม่เหล็ก F=BIL (หลักการมอเตอร์ไฟฟ้า) หมวดไฟฟ้า-แม่เหล็กขยับจาก 4→8 ข้อ (เพิ่มเป็น 2 เท่า) — post-deploy verification ยืนยัน 32 บทเรียน/68 ชุด/268 ข้อ ไม่มี id ซ้ำ ทุก field ครบ
37. เจาะลึกต่อตามลำดับความสำคัญ EE-at-3-พระจอมฯ ในข้อ 36 — ขั้นต่อไปคือ **A-Level คณิต 1 หมวดเวกเตอร์และจำนวนเชิงซ้อน** เพราะจำนวนเชิงซ้อนเป็นพื้นฐานตรงของการวิเคราะห์วงจรไฟฟ้ากระแสสลับ (แทนค่าอิมพีแดนซ์ด้วยจำนวนเชิงซ้อน) เพิ่มชุดใหม่ 1 ชุด (4 ข้อ): การคูณจำนวนเชิงซ้อน (3+2i)(1+i)=1+5i, สังยุคของ 4−3i, ขนาด (modulus) |3+4i|=5 (อธิบายว่าใช้สูตรเดียวกับขนาดอิมพีแดนซ์ใน AC), การบวกเวกเตอร์ (3,2)+(1,4) — ไม่ซ้ำกับ 4 ข้อเดิมในชุดเวกเตอร์/จำนวนเชิงซ้อน (ตรวจสอบด้วย Node ก่อนเขียน) A-Level คณิต 1 ขยับจาก 24→28 ข้อ (~93% ของ 30 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/69 ชุด/272 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมี 5 ตัวเลือกครบ
38. ทำงานต่อเนื่องตามลำดับความสำคัญ EE-at-3-พระจอมฯ — กลับไปที่ **TPAT3** (อันดับสูงสุดของลำดับ ยังมี % ครอบคลุมต่ำสุดในบรรดา 3 วิชาที่ให้ความสำคัญ ~63%) เจาะลึกหมวด "Mechanical & Physics Aptitude (Circuits & Matter)" ต่อยอดจากชุดมโนทัศน์เดิม (วงจรอนุกรม/ขนาน, สถานะสสาร) ด้วยชุดคำนวณจริง 1 ชุด (4 ข้อ): กฎของโอห์ม I=V/R, ความต้านทานรวมแบบอนุกรม (บวกตรงๆ), ความต้านทานรวมแบบขนาน (ผลรวมส่วนกลับ), ความหนาแน่น ρ=m/V — เลือกหัวข้อนี้เพราะเป็นพื้นฐานตรงที่สุดของการวิเคราะห์วงจรไฟฟ้าในบรรดาหมวดที่เหลือของ TPAT3 ทั้งหมด TPAT3 ขยับจาก 44→48 ข้อ (~69% ของ 70 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/70 ชุด/276 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมี 4 ตัวเลือกครบ (มาตรฐานถูกต้องสำหรับ TPAT3)

42. ผู้ใช้สั่งให้ทำงานต่อเนื่องอีกครั้ง — เจาะลึก TPAT3 หมวด "Scientific Thinking (Predict Experimental Outcome)" ต่อ (หมวดเดิมมีแค่ 1 ชุด/4 ข้อ) เพิ่มชุดที่ 2 (4 ข้อ): วงจรไฟฟ้า-ความสว่างหลอดไฟตามกฎของโอห์ม (I=V/R เชื่อมโยงกับลำดับความสำคัญ EE), การนำความร้อนของโลหะเทียบไม้, ขั้วแม่เหล็กเมื่อตัดแท่งแม่เหล็ก (ไม่มี magnetic monopole), การลอยตัวของลูกโป่งฮีเลียม (เปรียบเทียบความหนาแน่น) TPAT3 ขยับจาก 60→64 ข้อ (~91% ของ 70 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/74 ชุด/292 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมี 4 ตัวเลือกครบ
43. ผู้ใช้สั่งให้ทำงานต่อเนื่องอีกครั้ง — เจาะลึก TPAT3 หมวด "Science & Tech Literacy (Current Technology)" ต่อ (หมวดเดิมมีแค่ 1 ชุด/4 ข้อ) เพิ่มชุดที่ 2 (4 ข้อ) เลือกหัวข้อที่เชื่อมโยงตรงกับวิศวกรรมไฟฟ้า/อิเล็กทรอนิกส์ตามลำดับความสำคัญ EE: เซลล์แสงอาทิตย์ (photovoltaic effect ในสารกึ่งตัวนำ), แบตเตอรี่ลิเธียมไอออน (energy density สูงสำหรับ EV/อุปกรณ์พกพา), เครือข่าย 5G (ความเร็ว/latency/จำนวนอุปกรณ์เทียบ 4G), Internet of Things (การเชื่อมต่ออุปกรณ์กับอินเทอร์เน็ตอัตโนมัติ) TPAT3 ขยับจาก 64→68 ข้อ (~97% ของ 70 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/75 ชุด/296 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมี 4 ตัวเลือกครบ
44. ผู้ใช้สั่งให้ทำงานต่อเนื่องอีกครั้ง — เจาะลึก TPAT3 หมวดสุดท้ายที่ยังบางกว่าหมวดอื่น "Science & Tech Literacy (Historical Milestones)" เพิ่มชุดที่ 2 (4 ข้อ) เลือกเหตุการณ์ประวัติศาสตร์ที่เป็นรากฐานตรงของวิศวกรรมไฟฟ้าตามลำดับความสำคัญ EE: การค้นพบการเหนี่ยวนำแม่เหล็กไฟฟ้าของฟาราเดย์ (รากฐานเครื่องกำเนิดไฟฟ้า/หม้อแปลง), การประดิษฐ์ทรานซิสเตอร์ที่ Bell Labs 1947 (รากฐานอิเล็กทรอนิกส์สมัยใหม่), สงครามกระแสไฟฟ้า AC-DC ระหว่างเอดิสันกับเทสลา/เวสติงเฮาส์ (ทำไม AC ชนะ), สมการของแมกซ์เวลล์ (รวมไฟฟ้า-แม่เหล็ก-แสงเป็นทฤษฎีเดียว รากฐานการสื่อสารไร้สาย) TPAT3 ขยับจาก 68→72 ข้อ — **ครบทุกหมวดย่อยที่ทำได้แล้ว (9 หมวด × ~8 ข้อ เท่ากันหมด) จำนวนข้อรวมเกิน 70 ข้อจริงแล้ว** เพราะแต่ละหมวดมีชุดฝึกมากกว่าสัดส่วนจริงในบางหมวดเพื่อให้ฝึกซ้ำได้ ไม่ใช่การจำลอง 1:1 กับข้อสอบจริงอีกต่อไป — เหลือแค่มิติสัมพันธ์ (15 ข้อจริง) ที่ยังทำเองไม่ได้ (ใช้ลิงก์ภายนอกแทน) — post-deploy verification ยืนยัน 32 บทเรียน/76 ชุด/300 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมี 4 ตัวเลือกครบ **🎉 หลักไมล์ 300 ข้อทั้งระบบ**
45. ผู้ใช้สั่งให้ทำงานต่อเนื่องโดยไม่ต้องหยุดถาม — TPAT3 ครบทุกหมวดที่ทำได้แล้วจากข้อ 44 จึงย้ายไปเจาะลึก **A-Level ฟิสิกส์** ต่อตามลำดับความสำคัญ EE (อันดับ 2) เลือกหมวด "ฟิสิกส์แผนใหม่" ที่ยังมีแค่ 1 ชุด/4 ข้อ (บางที่สุดในบรรดา 4 หมวดที่เหลือ) เพิ่มชุดที่ 2 (4 ข้อ) เน้นหัวข้อที่เชื่อมโยงตรงกับอิเล็กทรอนิกส์/EE: ปรากฏการณ์โฟโตอิเล็กทริก (รากฐานเซลล์แสงอาทิตย์/โฟโตไดโอด), สารกึ่งตัวนำชนิด n (การเจือสาร), รอยต่อ p-n ในไดโอด (rectification), มวลพร่องกับ E=mc² — **พบข้อผิดพลาดระหว่างตรวจสอบ: เขียนตัวเลือกผิดเป็น 4 ข้อแทนที่จะเป็น 5 ข้อ (มาตรฐาน A-Level ฟิสิกส์) ในทั้ง 4 คำถามใหม่ — แก้ไขทันทีโดยเพิ่มตัวลวงข้อที่ 5 ที่สมเหตุสมผลให้ครบทุกข้อ** A-Level ฟิสิกส์ขยับจาก 24→28 ข้อ (~93% ของ 30 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/77 ชุด/304 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมี 5 ตัวเลือกครบ (แก้ไขแล้วยืนยันถูกต้อง)
46. ทำงานต่อเนื่องทันทีตามคำสั่ง — เจาะลึก A-Level ฟิสิกส์หมวด "คลื่นกลและแสง" ต่อ (ยังมีแค่ 1 ชุด/4 ข้อ) เพิ่มชุดที่ 2 (4 ข้อ) เลือกหัวข้อที่เชื่อมโยงกับวิศวกรรมโทรคมนาคม/สื่อสารตามลำดับความสำคัญ EE: การสะท้อนกลับหมดภายใน (Total Internal Reflection — หลักการของใยแก้วนำแสง), คลื่นแม่เหล็กไฟฟ้าเดินทางในสุญญากาศได้ (หลักการสื่อสารไร้สาย/ดาวเทียม), การแทรกสอดแบบหักล้าง (หลักการหูฟังตัดเสียงรบกวน), ดัชนีหักเห n=c/v A-Level ฟิสิกส์ขยับจาก 28→32 ข้อ (~107% ของ 30 ข้อจริง — เกินจำนวนข้อจริงแล้วเช่นเดียวกับ TPAT3) เขียนครบ 5 ตัวเลือกถูกต้องตั้งแต่แรกคราวนี้ (เรียนรู้จากข้อผิดพลาดในข้อ 45) — post-deploy verification ยืนยัน 32 บทเรียน/78 ชุด/308 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมี 5 ตัวเลือกครบ

65. **แก้บั๊ก UI: โจทย์ข้อยาวล้นออกนอกกรอบการ์ด** — ผู้ใช้ส่งภาพหน้าจอมาชี้ปัญหาในโหมดทำโจทย์ (`playQuiz`) ว่าข้อความโจทย์ที่ยาว (เช่น "1. Advertisement: ...") แสดงล้นออกนอกกรอบสี่เหลี่ยมมนของการ์ดคำถาม ทั้งที่ตัวเลือกคำตอบด้านล่างยังอยู่ในกรอบปกติ — **สาเหตุ:** โครงสร้างเดิมใช้ `<fieldset>` ครอบคำถามแต่ละข้อ แล้วใส่ข้อความคำถามใน `<legend>` ซึ่งเป็นพฤติกรรมมาตรฐานของ HTML ที่ browser จะวาด `<legend>` ให้ "คร่อม" เส้นขอบบนของ `<fieldset>` เสมอ (เหมือนป้ายชื่อกรอบฟอร์ม) — เมื่อข้อความยาวจนต้องขึ้นหลายบรรทัด ส่วนที่เกินจะทะลุออกนอกกรอบด้านบนแทนที่จะอยู่ในกรอบ เป็นข้อจำกัดของ `<legend>` เอง ไม่ใช่บั๊กจาก CSS ที่เขียนเอง — **วิธีแก้:** เปลี่ยน `<fieldset data-qidx="...">...<legend>...</legend>...</fieldset>` เป็น `<div data-qidx="...">...<p>...</p>...</div>` ธรรมดา (ตรวจสอบแล้วว่า `<fieldset>` ไม่ได้ถูกใช้เพื่อความสามารถพิเศษใดๆ ของฟอร์ม เช่น การจัดกลุ่ม radio ก็ทำผ่าน `name` attribute อยู่แล้ว ไม่ได้พึ่ง `<fieldset>`) พร้อมแก้ตัวเลือก selector ที่ใช้หา element ตอนตรวจคำตอบ (`player.querySelector('fieldset[data-qidx=...]')` → `'div[data-qidx=...]'`) — post-deploy verification: grep ยืนยันไม่มี `fieldset`/`legend` หลงเหลือในไฟล์เลย + เรียก `playQuiz()` จริงผ่าน Node กับข้อมูลจริงชุดแรกของระบบ ยืนยันว่า render ได้ปกติ ไม่มี error, มี `<div data-qidx="0">`, ข้อความคำถามแสดงครบ — ไม่กระทบข้อมูล quiz/lesson เลย (เนื้อหาเดิมทั้งหมดไม่เปลี่ยน)
66. ผู้ใช้สั่งให้ทำงานต่อเนื่องจนครบทุกวิชาไม่ต้องหยุดถาม — เริ่มจาก **A-Level อังกฤษ** (ต่ำสุด 67%) ตามที่ตกลงกันไว้ เพิ่มชุดที่ 2 ให้ครบทั้ง 8 หมวดที่ยังบางในรอบเดียว (32 ข้อ): การอ่านจับใจความ (คนขายดอกไม้), Text Completion (as soon as/in order to/even though/unless), Paragraph Ordering (สมัครงาน/ชงชา/มลพิษ/สุนัขหาย — สลับ label ไม่ให้ตอบ A-B-C-D ตรงๆ ทุกข้อ), Advertisement & Review (โฆษณายิม + รีวิวหูฟัง), News & Article (ข่าวทุนการศึกษา + บทความการนอน), Graph/Chart Data (สำรวจการเดินทาง + ค่าใช้จ่ายครัวเรือน), Error Identification (each of+เอกพจน์, the number of+เอกพจน์, double comparative, past perfect tense), Best Headline Selection (ข่าวถุงพลาสติก/หุ่นยนต์โรงเรียน/ราคากาแฟ/กบสายพันธุ์ใหม่) — เขียนและ verify ผ่านตั้งแต่ครั้งแรกไม่มี error เลยทั้ง 8 ชุด (เพราะ grep หา id ทั้งไฟล์ก่อนตั้ง id ใหม่ทุกครั้งตามบทเรียนจากรอบก่อนๆ) A-Level อังกฤษขยับจาก 40→72 ข้อ (90% ของ 80 ข้อจริง, **เกินเพดานที่ทำได้จริง 60 ข้อไปแล้ว**) **ครบทุกหมวดย่อยที่ทำได้แล้วทั้ง 9 หมวด (8 ข้อเท่ากันหมด)** เหลือแค่ฟัง-พูดที่บล็อกถาวร — post-deploy verification ยืนยัน 32 บทเรียน/110 ชุด/436 ข้อ ไม่มี id ซ้ำ ทุก field ครบ
67. ทำงานต่อเนื่องทันทีตามคำสั่ง — ย้ายไปเจาะลึก **TGAT2** ต่อ (75%) เพิ่มชุดที่ 6 ให้ครบทั้ง 3 หมวด: Verbal (ใจกว้าง↔ใจแคบ, สัตว์เลี้ยงลูกด้วยนม vs งู, นักเขียน:หนังสือ, ชาญฉลาด~ปราดเปรื่อง), Numerical (ผลรวม-ผลต่างสองจำนวน, ความน่าจะเป็นแบบไม่ใส่คืน, ค่าที่หายไปจากค่าเฉลี่ย, เปอร์เซ็นต์ซ้อนส่วนลด+ภาษี), Logical (contrapositive, เปรียบเทียบราคา 4 ฝ่าย, ปริศนาจับคู่อาชีพ 4 คน — ตรวจสอบความเป็นเอกลักษณ์ของคำตอบก่อนสรุป เพราะร่างแรกมีทางแก้ได้ 2 แบบ ปรับเงื่อนไขให้ตัดสินได้คำตอบเดียวก่อนเขียนจริง, exclusive or) TGAT2 ขยับจาก 60→72 ข้อ (90% ของเพดานที่ทำได้จริง 80 ข้อ) TGAT รวมขยับจาก 152→164 ข้อ (82% ของ 200 ข้อจริง) — post-deploy verification ยืนยัน 32 บทเรียน/113 ชุด/448 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมี 5 ตัวเลือกถูกต้อง (มาตรฐาน TGAT2)
68. ผู้ใช้สั่งให้ทำ **TGAT1 และ A-Level คณิต 2 ให้ได้ 100%** โดยเฉพาะ — **TGAT1**: เพิ่มชุดที่ 5 ให้ครบทั้ง 3 หมวด (12 ข้อ): Question-Response (ปฏิเสธคำเชิญอย่างสุภาพ, ขอข้อมูลเพิ่มเติม, เสนอความช่วยเหลือ, ปลอบใจข่าวไม่ดี), Text Completion (too...to, reported speech will→would, would rather+กริยาเปล่า, present perfect continuous), Reading Comprehension (เรื่องใหม่ครูสอนภาษาอังกฤษ Ploy) TGAT1 ขยับจาก 48→60 ข้อ **= 100% ของ 60 ข้อจริงพอดี** — **A-Level คณิต 2**: เพิ่ม 2 ชุดใหม่ในหมวดที่ยังไม่เคยเจาะลึกซ้ำ (8 ข้อ): "ลำดับ ดอกเบี้ย สถิติ และความน่าจะเป็น" (ผลรวมอนุกรมเลขคณิต, ดอกเบี้ยแบบธรรมดา, ฐานนิยม, ความน่าจะเป็นหลายรสชาติ) และ "จำนวนและพีชคณิต" (ลำดับเรขาคณิต, ฟังก์ชันประกอบย้อนทิศทาง g(f(x)), อินเตอร์เซกชันเซต, ค่าความจริง OR) — **พบและแก้ข้อผิดพลาดระหว่างตรวจสอบ: คำถามความน่าจะเป็นลูกอมเขียนตัวเลือกผิดเป็น 4 ข้อแทนที่จะเป็น 5 ข้อ (มาตรฐาน A-Level คณิต 2) — เพิ่มตัวเลือกที่ 5 (3/4) แล้ว verify ซ้ำจนผ่าน** เนื่องจากขนาดข้อสอบจริง (30 ข้อ) ไม่หารด้วย 4 ลงตัว จึงปัดขึ้นเป็น 32 ข้อ A-Level คณิต 2 ขยับจาก 24→32 ข้อ **= 107% ของ 30 ข้อจริง (เกิน 100%)** — post-deploy verification ยืนยัน 32 บทเรียน/118 ชุด/468 ข้อ ไม่มี id ซ้ำ ทุก field ครบ ทุกคำถามมีตัวเลือกถูกต้องตามมาตรฐานวิชานั้นๆ **ทั้ง TGAT1 และ A-Level คณิต 2 บรรลุเป้าหมาย 100%+ ตามที่ผู้ใช้สั่งแล้ว**
69. ผู้ใช้สั่งให้เพิ่ม **A-Level เคมี ให้ได้ 90%** (วิชาเดียวที่เหลืออยู่พอดี 80% ไม่เกิน) — เลือกเจาะลึกหมวด "เคมีไฟฟ้า" (เชื่อมโยงกับหลักการแบตเตอรี่/วิศวกรรมไฟฟ้าได้ดีที่สุดในบรรดาหมวดที่เหลือ) เพิ่มชุดที่ 2 (4 ข้อ): ความแตกต่างเซลล์อิเล็กโทรไลต์ vs กัลวานิก (ทิศทางพลังงานตรงข้ามกัน), นิยามตัวรีดิวซ์ (ให้อิเล็กตรอน ตัวมันเองเกิดออกซิเดชัน), อนุกรมกิจกรรมโลหะ (โลหะ active มากออกซิไดซ์ง่ายกว่า), การชุบโลหะด้วยไฟฟ้า (ตัวอย่างเซลล์อิเล็กโทรไลต์ในชีวิตจริง) — เนื่องจากขนาดข้อสอบจริง (35 ข้อ) ไม่หารด้วย 4 ลงตัว เช่นเดียวกับคณิต 2 ก่อนหน้า จึงปัดขึ้นเป็น 32 ข้อ A-Level เคมีขยับจาก 28→32 ข้อ **= 91% ของ 35 ข้อจริง (เกิน 90% ตามที่สั่ง)** เขียนครบ 5 ตัวเลือกถูกต้องทุกข้อตั้งแต่ครั้งแรก — post-deploy verification ยืนยัน 32 บทเรียน/119 ชุด/472 ข้อ ไม่มี id ซ้ำ ทุก field ครบ **ทั้ง 8 วิชาอยู่ที่ 90%+ ของขนาดข้อสอบจริงแล้วทุกวิชา**
70. **ขยายฟีเจอร์สรุปบทเรียน: คำอธิบายเห็นภาพขึ้น + ตัวอย่างจาก 2 เป็น 5 ข้อ/บท** — ผู้ใช้ขอให้ `lesson.detail.explanation` ละเอียด/เห็นภาพมากขึ้นแต่ยังง่ายต่อการเข้าใจ และเพิ่ม `examples[]` จาก 2 เป็น 5 ข้อต่อบทเรียน — เขียนคำอธิบายใหม่ทั้ง 32 บทด้วยรูปแบบ "ลองนึกภาพ..." เปิดด้วยภาพเปรียบเทียบที่จับต้องได้ก่อนเสมอ (เช่น ไม้กระดก→คานงัด, เครื่องหยอดเหรียญ→ฟังก์ชัน, ท่อน้ำมีปั๊ม→วงจรไฟฟ้า V=IR, ก้อนหิมะกลิ้งลงเขา→ดอกเบี้ยทบต้น, โดมิโน→การให้เหตุผลนิรนัย, สวิตช์ไฟบ้าน→เกตตรรกะดิจิทัล) แล้วโยงเข้าสูตร/หลักการเดิมโดยไม่ตัดเนื้อหาเดิมออก — เพิ่มตัวอย่างใหม่ 3 ข้อ/บท (รวม 96 ข้อทั้งระบบ) ต่อจากตัวอย่างเดิม 2 ข้อ ให้ครบ 5 ข้อ/บท คำนวณตรวจทานเองทุกข้อก่อนใส่ — ใช้สคริปต์ Node bulk-edit แบบเดิม (eval ไฟล์ทีละไฟล์ หา lesson ตาม id แล้วแก้เฉพาะ `detail.explanation` และต่อ array `detail.examples` โดยไม่แตะ `detail.concept`/`body`/`formula`/quiz ใดๆ เลย) — post-deploy verification เพิ่มเงื่อนไขใหม่ให้สคริปต์ตรวจสอบ: ทุกบทต้องมี `examples.length === 5` พอดี (ยืนยัน 0 ปัญหาทั้ง 32 บท) และยืนยันว่าจำนวนชุดข้อสอบ/ข้อสอบทั้งระบบไม่เปลี่ยนแปลงเลย (119 ชุด/472 ข้อ เท่าเดิมทุกตัวเลข ยืนยันว่าไม่กระทบเนื้อหาข้อสอบ) — เรียก `renderLessons()` จริงผ่าน Node ยืนยันไม่ throw error และ HTML ที่ได้มีเนื้อหาครบ (127,064 ตัวอักษร เพิ่มขึ้นจาก 81,503 สะท้อนเนื้อหาที่ขยายจริง)
71. **แก้ข้อความ footer ที่คลาดเคลื่อนจากความจริง** — ผู้ใช้ถามว่าข้อความ "prototype" ในหน้า UI เอาออกได้หรือยัง ตรวจสอบ 360° รอบ UI ทั้งหมดที่มีคำว่า "ตัวอย่าง"/"prototype" พบว่า **ยังเอาออกไม่ได้ทั้งหมด** เพราะ `compareData[]` (ตารางเปรียบเทียบสัดส่วนคะแนน 12 มหาวิทยาลัย) ยังเป็นข้อมูลแต่งขึ้นจริง (สังเกตได้จาก `gpax:20` ซ้ำกันทุกมหาวิทยาลัยและชุดตัวเลข tpat3/alevel ที่ซ้ำกันหลายที่) และ `news[]` ยังมีข้อความ "ประกาศ**ตัวอย่าง**..." ระบุตรงๆ ในตัวข้อมูลเอง — **แต่พบจุดที่คลาดเคลื่อนจริง**: footer เดิมเขียนว่า "ข้อมูลกำหนดการและเกณฑ์คะแนนเป็นตัวอย่างสำหรับออกแบบ UI เท่านั้น" ทั้งที่ `rounds[]`/`examDates[]` (กำหนดการ) ถูกแทนที่ด้วยปฏิทิน TCAS70 จริงที่ค้นคว้า+ตรวจสอบไขว้แล้วตั้งแต่ข้อ 14 ของ changelog นี้ — footer จึงพูดเกินจริงว่ากำหนดการเป็นแค่ตัวอย่างด้วยทั้งที่ไม่ใช่แล้ว แก้ข้อความใหม่ให้ระบุเฉพาะ "ตารางเปรียบเทียบเกณฑ์คะแนนรายมหาวิทยาลัยและประกาศข่าว" ว่าเป็นตัวอย่าง พร้อมระบุชัดว่ากำหนดการ TCAS70 เป็นข้อมูลค้นคว้าจริงแล้ว — ตรวจสอบ `mockBanner` (แบนเนอร์เตือนหลักในหน้า Dashboard/Calendar) แยกต่างหากแล้วพบว่าเขียนแยกถูกต้องอยู่แล้ว (บอกชัดว่ากำหนดการค้นคว้าจริงแล้วแต่ตารางเปรียบเทียบ+ข่าวยังเป็นตัวอย่าง) ไม่ต้องแก้ — post-deploy verification: grep ยืนยันไม่มีข้อความเก่าหลงเหลือ + รันฮาร์เนสเต็มรูปแบบผ่านปกติไม่กระทบส่วนอื่น (เป็นการแก้ static text นอก script หลัก)

72. **"ประกาศล่าสุด" เปลี่ยนจากข้อมูลตัวอย่างเป็นข่าวจริง + เพิ่มลิงก์ไปหน้าข่าวทางการ** — ผู้ใช้ถามว่าทำอย่างไรให้ส่วน "ประกาศล่าสุด" เป็นข้อมูลปัจจุบันได้จริง อธิบายว่าแอปนี้เป็น static file แจกจ่ายแบบ zip ไม่มี backend/API จึงดึงข่าวสดอัตโนมัติไม่ได้จริงทางสถาปัตยกรรม เสนอ 3 ทางเลือก (ลิงก์ออกอย่างเดียว / ค้นคว้าฝังข้อมูลจริงครั้งเดียว / ทำทั้งสองอย่าง) ผู้ใช้เลือก **"ทำทั้งสองอย่างพร้อมกัน"** — ค้นคว้าข่าว TCAS70 จริงผ่าน WebSearch+WebFetch หลายแหล่ง ลอง `mytcas.com/news/` ตรงๆ ก่อนแต่พบวันที่ย้อนหลังผิดยุค (ดูเหมือนรอบ TCAS69) จึงไม่ใช้ ข้ามไปยืนยันผ่าน `eduzones.com` (มีวันที่เผยแพร่ชัดเจนตรวจสอบได้) ได้ข่าวจริง 3 รายการ: (26 เม.ย. 69) ทปอ. แถลงนโยบาย TCAS70 ฉบับเต็ม (สอบฟรีทุกวิชา, เปลี่ยนเป็น PBT ทั้งหมด, ระบบ TCASfolio+TCAS Verified ใหม่), (26 มิ.ย. 69) SIIT ธรรมศาสตร์ เปิดทุน OSP รอบ Portfolio กว่า 100 ทุนสำหรับสายวิศวกรรม (เชื่อมโยงกับกลุ่มเป้าหมาย EE ของแอปนี้โดยตรง), (9 ก.ค. 69) MyTCAS ย้ำการลงทะเบียนเป็นขั้นตอนแรกที่ต้องทำก่อนสมัคร TGAT/TPAT — แทนที่ `news[]` เดิม 4 รายการที่แต่งขึ้นทั้งหมดด้วยข่าวจริง 3 รายการ พร้อม field `source` (URL อ้างอิง) ต่อรายการ — แก้ `renderNews()` เอา badge "ตัวอย่าง" ที่ hardcode ทุกรายการออก เปลี่ยนเป็นลิงก์ "[แหล่งข่าว]" ต่อท้ายแต่ละข่าวแทน — เพิ่มปุ่มลิงก์ภายนอก "ดูข่าวล่าสุดทั้งหมด →" ไปยัง `mytcas.com/news/` ตรงๆ (แยกจากปุ่ม "ไปหน้าลิงก์ทางการ →" เดิมที่พาไปแท็บปฏิทินภายในแอป) — ลบ label "(ตัวอย่าง)" ออกจากหัวข้อ "ประกาศล่าสุด" และแก้ข้อความ disclaimer ใต้รายการข่าวให้บอกตามจริงว่าเป็นการคัดเลือก+ตรวจสอบไขว้ด้วยมือ ไม่ใช่ระบบดึงข่าวอัตโนมัติ — แก้ `mockBanner` ให้แยกชัดว่า "ประกาศข่าว" ไม่ได้เป็นข้อมูลตัวอย่างแล้ว เหลือแค่ตารางเปรียบเทียบมหาวิทยาลัย (`compareData`) ที่ยังเป็นตัวอย่างอยู่ — post-deploy verification: รันฮาร์เนสเต็มรูปแบบยืนยัน 32 บทเรียน/119 ชุด/472 ข้อ เท่าเดิมทุกตัวเลข (ไม่กระทบข้อมูลข้อสอบ/บทเรียนเลย เพราะแก้เฉพาะ `news[]`/`renderNews()`/ข้อความ static)

73. **เอา "prototype" ออกจากป้าย badge ในหัวเว็บ** — ผู้ใช้ส่งภาพหน้าจอชี้ป้าย `rev. prototype` มุมขวาบนของ header ขอให้เอาคำว่า prototype ออก — grep ทั้งไฟล์ยืนยันว่าเป็นจุดเดียวที่มีคำนี้เหลืออยู่ในหน้า UI (สอดคล้องกับ §13 ข้อ 71 ที่เคยแก้ footer ไปแล้ว) เปลี่ยนข้อความ badge จาก `rev. prototype` เป็น `beta` (คงความหมาย "ยังพัฒนาต่อเนื่องอยู่" ไว้ แต่ไม่สื่อว่าเป็นแค่ demo ทั้งที่เนื้อหาจริงส่วนใหญ่ค้นคว้า+ตรวจสอบแล้ว) — post-deploy verification: grep ยืนยันไม่มีคำว่า prototype เหลือในไฟล์เลย + รันฮาร์เนสเต็มรูปแบบยืนยัน 32 บทเรียน/119 ชุด/472 ข้อ เท่าเดิม (แก้แค่ static text ใน header ไม่กระทบข้อมูลใดๆ)
